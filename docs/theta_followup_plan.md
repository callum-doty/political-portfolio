## 0. A correction found while starting this document — test this first

While preparing to write up "guess #2" (the environment-asymmetry hypothesis) below, checking the actual code in `scripts/solve_bellman_lsm.py` turned up something that changes the priority order: **neither of the two calibrated mechanisms this session spent most of its effort on — $\hat\eta(\text{tier})$ and $\sigma_G(\Delta t)$ — actually affected the reported result.**

- `eta_by_tier` is passed into `run_lsm()` and stored in the output dict for reporting, but is never multiplied into anything inside the function. This is not a subtle bug — it is structurally impossible for $\eta$ to matter under the current "wait" branch design, because $D_{i,t}$ never changes while waiting (no discretionary spending happens), so $\eta\cdot\Delta D_{i,t}=\eta\cdot 0=0$ regardless of $\hat\eta$'s value. The "two $\eta$-brackets" reported in §8.2 differed only in `resid_std_by_tier` (the *residual* noise scale from each cycle's regression — opponent spending variation *unexplained* by reaction), not in $\eta$ itself.
- $G_t$ was never simulated in `solve_bellman_lsm.py` at all — no `g_paths`, no reference to `SIGMA_G_PER_SQRT_DAY`. This is *consistent* with §5.5's scope boundary (never fold $\Delta G$ into $\mu_i$, since $\alpha_3$ was never estimated for within-cycle GB movement) but it means the national-environment calibration was completely absent from this run, not just present-but-symmetric as an earlier explanation loosely suggested.

**What this means:** the "deploy now" result was driven entirely by two things — the idiosyncratic $\varepsilon_{i,t}$ decay process and opponent's *unexplained* residual spending noise — both symmetric, mean-zero processes. The two mechanisms most likely to introduce a genuine asymmetry or a real reactive cost to spending were never in play. This is arguably a more important thing to fix than either guess below, because until it's fixed, guesses 1 and 2 are being tested on an incomplete model of $P$.

### 0.1 Implementation steps

1. **Give $\eta$ something to react to.** The "wait" branch currently means "zero discretionary spending, ever, until a deploy decision fires." To let $\eta$ matter, the wait branch needs a small, non-discretionary baseline spending trickle it can react to — the most defensible source is real, ongoing candidate-committee disbursement growth (already tracked as `cand_d_total`, currently held perfectly fixed as the floor). Model the floor itself as slowly growing over the remaining periods (a simple rate estimated from the historical panel), and let $R_{i,t}$ react to *that* via $\eta(\text{tier})$, in addition to the residual noise already modeled.
2. **Wire $\hat\eta$ into the deploy branch too.** `_deploy_value()` calls the LP allocator (`optimize()`), which has no adversarial-response parameter at all — unlike `optimize_nonlinear()`, which already supports `eta` for exactly this purpose (§I of the full derivation doc; `allocator.py`'s `_reactive_r`). Either switch the deploy-branch evaluation to a fast approximation that includes the same reactive-$R$ mechanic, or add an equivalent linear adjustment to the LP formulation (reduce the effective MSG gradient by a factor reflecting $\eta$, matching `_msg_vec`'s `eta`-adjusted gradient in the nonlinear path).
3. **Bring $G_t$ into the simulation, respecting the scope boundary.** The boundary says: don't apply $\alpha_3$ to within-cycle $G_t$ movement in the *margin model*. It does not say $G_t$ can't affect the *basis features* or the *value comparison* directly. One option: add $G_t$ (or its realized path) as a fifth regression-basis feature, so the continuation-value regression can learn a $G_t$-dependent adjustment empirically, without ever touching $\mu_i$'s structural formula. This sidesteps the boundary rather than violating it.
4. **Re-run the full $K=2000$, two-bracket LSM with both fixes applied**, and check whether $\Theta(0)$ changes. If it still favors full deployment, that is a stronger, more complete result than the current one (since it would no longer be attributable to two calibrated mechanisms sitting idle). If it changes, the current §8.2 finding needs to be revised, not just footnoted.

## 1. Guess A: the binary deploy/hold framing is too coarse

*Rescoped 2026-07-17, superseding the original sketch below §1.1 — the first pass (grid search over $\phi$ with a single-shot LP call per grid point, no carried-forward state) was structurally insufficient in two ways caught before implementation started: it would have masked concavity behind a linear interpolation, and it silently assumed the unspent fraction could be forfeited rather than carried forward. Both are corrected in §1.3 below. Not yet implemented — this is the design, not a result.*

### 1.1 The hypothesis

The current backward induction only ever compares two extremes at each period: deploy the entire remaining $F_t$, or hold all of it. A real committee's actual discretion is continuous — spend some fraction now, reserve the rest. It is possible that "hold 100%" genuinely loses to "deploy 100%" today, while some smaller reserve (say 20-30%) would beat both, and the current framework simply cannot express that middle option, so it never gets to win the comparison. The `eta_fit_2024` bracket's $\Theta(0)=-0.039$ (§4, §8.2) — an order of magnitude closer to indifference than `eta_fit_2022`'s $-0.517$ — is the concrete, data-grounded reason to expect this generalization might actually locate a non-zero interior answer rather than just re-confirm the corner. §7's finding sharpens why: $\Theta(t)$ as currently defined measures the value of flexibility *net of* what the deploy branch's own variance-convolution already captures for free, and that "already captured" share depends on how much of $F_t$ is being committed — so a coarser or finer split of the same total commitment could plausibly move the answer even without changing $F_0$ itself.

### 1.2 Why a naive grid search over $\phi$ would not work

Two structural requirements, both missed by the original sketch, have to hold simultaneously or this generalization will silently reproduce the binary answer while looking continuous:

1. **The deploy-value-at-$\phi$ function must be concave in $\phi$, not linearly interpolated between the two corner values.** $\Theta_\phi=\phi\, V^{\text{deploy}}+(1-\phi)V^{\text{wait}}$ is a convex combination of two fixed scalars — maximizing a linear function over a compact interval $[0,1]$ is mathematically guaranteed to land on a boundary unless the function happens to be exactly flat. That formulation could never surface an interior optimum; it would just recompute the existing binary answer with extra steps. The genuine source of concavity is the shadow price of capital falling as more is committed: `margin_gradient()`'s $c\cdot(1/D-(1+\eta)/T)$ term (`solve_bellman_lsm.py:187`) is strictly decreasing in $D$ — the same mechanism already verified as the structural driver behind §7's variance-convolution finding. Capturing it requires actually re-solving the LP allocator at each candidate deployment level, not interpolating between endpoints.
2. **Unspent capital must be carried forward, not forfeited.** Since `margin_gradient` stays positive for essentially every realistic $(D,T)$ pair in this universe (races here are heavily outspent, so $D\ll T$ almost everywhere), permanently setting money aside and never spending it is always weakly dominated by spending it eventually — a one-time static split chosen once at $t=0$ and never revisited would therefore never rationally choose $\phi<1$. This has to be a genuine sequential (impulse-control) problem: the reserve $F_t$ becomes a state variable that future periods can still act on, not a single interpolated choice at $t=0$. In the finance analogy this is closer to a multiple-exercise Bermudan option than a single European-style choice.

### 1.3 Implementation steps

1. **Discretize the reachable budget grid.** $\mathcal F=\{0,0.25F_0,0.5F_0,0.75F_0,F_0\}$ as a starting resolution (5 points). Legal actions from state $f\in\mathcal F$ are $a\in\mathcal F$ with $a\le f$; the post-action state carried into the next period is $f-a$.
2. **Fix the floor-reset bug this design would otherwise inherit.** `_deploy_value()` currently hardcodes `d_t = floor_arr.copy()` (`solve_bellman_lsm.py:299`) — the *original* candidate floor, on every call, regardless of what a path has already deployed. Calling the LP repeatedly against this same fixed baseline would give the same linearized `msg_i` every time (the existing LP-vs-nonlinear investigation, §8.2, already established that `optimize()` treats `msg_i` as a frozen constant per call, with no within-call diminishing-returns mechanism — it degenerates to a greedy knapsack). Without this fix, re-running the LP at successive nodes would trace out a flat line, not a concave curve, and the whole exercise would silently fail to find anything but a corner for a reason having nothing to do with economics. **Required change**: `d_t` must become `floor_arr + cumulative_deployed_state`, so each successive call re-linearizes `margin_gradient` at the higher, already-topped-up $D$ — this is what makes the sequence of LP calls trace out the true concave value-of-budget curve via successive re-linearization, consistent with how the existing chain-rule approximation already works within a single call.
3. **Backward induction over the expanded state**, for each state $(t,\omega,f)$ and each legal action $a$:
   $$V_t(\omega,f)=\max_{a\in\mathcal A(f)}\Big[\text{LP\_Alloc}(a,\mathbf X_t(\omega),\ \text{floor}=\text{floor}_0+F_0-f+a)+\hat{\mathbb E}_t\big[V_{t+1}(\omega,f-a)\big]\Big]$$
4. **Continuation regression, conditioned on remaining budget.** Two options, not mutually exclusive:
   - *Unified*: add $f'$ and $(f')^2$ as a sixth and seventh feature to the existing compressed basis ($E[\text{Seats}]_t$, $\text{Var}[\text{Seats}]_t$, $\text{MaxMSG}_t$, near-threshold count, $G_t$), one joint regression per period. Preferred as the starting approach — pools information across budget states, which matters because endogenous action choice (rather than uniform grid sampling) will likely leave some $(t,f)$ cells thin, especially in early periods before paths have had a chance to spread across reserve levels.
   - *Grid-stratified*: a separate regression per reachable $f'$ per period — avoids imposing a quadratic functional form on the budget/political-feature interaction, at 5x the (computationally trivial) regression cost. Use as a diagnostic check on the unified fit's residuals, not the default, given the sparse-cell risk above.
5. **Compute cost, using this project's actual $N_{\text{periods}}=7$ (live run), not an illustrative $T=15$.** Sum of legal actions across a 5-point grid is $1+2+3+4+5=15$ per (path, period); at $K=2000$: $15\times2000\times7=210{,}000$ LP calls $\times\sim$11ms $\approx$ 38 minutes per bracket. This is a batch job, not a quick script, but tractable.
6. **Grid-resolution sensitivity is not optional, given how close to indifference the target is.** A 5-point (25%-step) grid could report a corner as optimal purely from resolution if the true interior optimum is narrow (e.g. a 5–10% reserve) — indistinguishable, from the output alone, from a genuine bang-bang result. Run the 5-point grid first; if it reports anything other than a clean corner, re-run at an 11-point (10%-step) grid to confirm the interior answer isn't a grid artifact. Cost scales roughly as the triangular sum of grid points, so 5→11 points is roughly a 4.4x multiplier ($\approx$170 minutes per bracket) — worth budgeting for upfront rather than discovering after a coarse run comes back at a corner with no way to tell if that's real.
7. **Report the optimal $\phi^*(t)$ schedule** (or, more precisely, the optimal action $a^*$ from each reachable state), not just a binary decision — this is a strictly more informative output than the current binary one regardless of what it finds, and directly answers "how much should the campaign hold back today," not just "should it hold back everything or nothing."

## 2. Guess B: the current environment may be asymmetrically favorable

### 2.1 The hypothesis

At D+5.02 on the generic ballot, the current environment is unusually good for Democrats relative to most of the historical range in the four-cycle series (2018 D+8.6, 2020 D+7.0, 2022 D$-$1.0, 2024 D$-$1.2 — recall these are *final* cycle values from Paper I's `generic_ballot_by_cycle.csv`; the relevant comparison here is against the *historical daily trajectory's* typical range, not just the four endpoints). If the true process is mean-reverting toward a lower long-run level rather than a pure zero-drift random walk, waiting risks a one-sided loss (the environment fading back toward its historical average), which would argue *for* locking in current conditions by deploying now — but this is a directional effect a symmetric, zero-drift process cannot express at all, whichever direction it points.

**Note the honest complication already surfaced in §0**: since $G_t$ was not in the tested model at all, this hypothesis cannot yet explain the *actual* result reported in §8.2 — it can only be tested once §0's fix (bringing $G_t$ into the analysis) is done. Sequence this after §0, not instead of it.

### 2.2 Implementation steps

1. **Fit an Ornstein-Uhlenbeck model *with drift* to the pooled four-cycle historical series**, not just its volatility (§5.3 already fit the volatility term structure but explicitly deferred formal RW-vs-OU model selection). Regress $\Delta G_t$ on $(t\text{-time})$ and current level $G_t$: the standard discretized-OU regression is $G_{t+\Delta}-G_t = \kappa\Delta(\bar G - G_t) + \text{noise}$, i.e. regress $\Delta G$ on $G_t$ (with a constant) to recover $\kappa$ and $\bar G$.
2. **Check where D+5.02 sits relative to the fitted $\bar G$.** If $\bar G$ is meaningfully below +5.02, quantify the implied expected drift over the remaining ~98 days: $\mathbb E[\Delta G] \approx \kappa\cdot(\bar G - 5.02)\cdot(\text{days remaining})$.
3. **Re-run the LSM (after §0's fix) with this asymmetric, mean-reverting $G_t$ process** in place of a zero-drift assumption, and check whether $\Theta(0)$ or the optimal deployment fraction (if §1 is also implemented) changes, and in which direction. Note the direction is not obvious in advance: a expected-downward drift could *strengthen* the case for deploying now (lock in today's favorable conditions before they fade) rather than weaken it — this hypothesis needs to be tested, not assumed to point toward more patience.
4. **Sanity-check against the historical series directly**: for each of the four historical cycles, identify the point (if any) where that cycle's trajectory was at a similarly favorable level to today's +5.02, and check what actually happened to the environment between that point and Election Day in that cycle. With only 4 cycles this is illustrative, not a statistically powered test, and should be reported as such.

## 5. Reconciling two calibration-reliability caveats (added 2026-07-17)

Reviewing `outputs/eta_reaction_by_tier_fig.png` and `outputs/gb_asymmetry_check_fig.png`, two caveats already on record in Paper III §5.5 had never been resolved: $\hat\eta(\text{tier})$'s leave-one-cycle-out instability, and $\sigma_G$'s same-direction late-cycle realized moves sitting in apparent tension with this session's OU-drift fit (§0.1 above). Both are now reconciled with a formal check (`scripts/reconcile_eta_sigma_g_instability.py`), not just re-described.

**$\hat\eta(\text{tier})$ instability — first pass said "mostly noise"; that pass was underpowered, and a properly powered second pass says otherwise.** First pass: checking whether each tier's 2022 and 2024 95% CIs overlap (rather than comparing point estimates alone) found **5 of 7 tiers overlap**, including Toss-Up, Lean D, and Lean R — suggesting their large point-estimate swings (Toss-Up: 0.73 vs. 0.34) were honestly *unresolved*, not confirmed real. **A single pairwise comparison has very little power, so this was a falsely reassuring answer, not a validated one.**

Second pass (2026-07-17, same session, extending to all 7 cycles this repo has usable dated IE data for — 2012 through 2024; per §4.2's own data-quality table, the older cycles have *better* date completeness than 2022/2024, not worse): fitting eta per cycle and running one joint test per tier gives real statistical power, and the answer changes. **5 of 7 tiers show statistically significant cycle-to-cycle variation** (Safe D $p=0.002$, Lean D $p<0.001$, Toss-Up $p=0.001$, Likely R $p<0.001$, Safe R $p<0.001$) — including **Toss-Up and Lean D, the two tiers this project leans on hardest**. Toss-Up's single-cycle estimate actually goes negative in 2016 ($-0.22$), a sign flip the 2-cycle check could never have shown. Only Likely D ($p=0.06$) and Lean R ($p=0.22$, thinnest data of any tier) aren't confirmed to vary — plausibly still a power limitation for Lean R, not genuine stability.

**Net effect, superseding the first pass: this is not a narrow Safe-D/Safe-R problem — it is a whole-table problem.** Precision within a given 2-cycle pool (§4.4's activity-rate argument) is a different claim from temporal stability across cycles, and most tiers fail the stability test once tested with real power. The pooled point estimates remain the best available and are not withdrawn, but should be read as averages over confirmed real variation, not fixed constants. See Paper III §5.5's twice-updated Validation C.

**$\sigma_G$'s late-cycle bias — real direction, unreliable magnitude, not in conflict with the OU fit.** Extending Validation B's Sept-1-to-Election-Day check from 2 cycles to all 4 (2018, 2020, 2022, 2024): **all four cycles moved toward Republicans** in that window (not just the 2 originally checked), but the size of the move varies enormously — 2018/2020 were tiny ($z\approx-0.05$ to $-0.11$, indistinguishable from noise) while 2022/2024 were substantial ($z\approx-1.3$ to $-1.4$). Mean $z$ across all four is $-0.73$, materially smaller than the $-1.37$ the 2-cycle check alone suggested. This is **not in conflict** with the OU-drift fit's "no significant level mean-reversion" finding (§0.1) — the two ask different questions: the OU fit pools all lags across the full trajectory and asks whether $G_t$ reverts toward a fixed $\bar G$ regardless of calendar position; this check asks specifically about the late-cycle window. A weak, inconsistent-magnitude late-cycle drift can coexist with no detectable level mean-reversion overall. **Practical implication, if this pattern is real**: the consistent direction (toward Republicans, i.e. away from today's Democratic-favorable D+5.02) argues for *locking in* the current environment by deploying now, not for patience — reinforcing, not reversing, §8.2's "deploy now" finding, for a reason distinct from anything tested in §0 or §2's OU fit. Given the wildly inconsistent magnitude across only 4 data points, this should be read as a real but weak and unreliable signal, not a basis for a new calibrated drift term.

All reconciliations are saved (`outputs/eta_instability_reconciliation.csv`, `outputs/sigma_g_late_cycle_reconciliation.csv`, `outputs/eta_seven_cycle_extension.csv`, `outputs/eta_seven_cycle_joint_test.csv`) and reflected in Paper III §5.5.

## 6. From "eta is unstable" to a usable uncertainty-aware model (added 2026-07-17)

§5 established that $\hat\eta$ genuinely varies cycle-to-cycle for 5 of 7 tiers. This section turns that finding into two concrete deliverables, chosen from a wider brainstorm (hierarchical Bayesian, hidden regimes, state-space/Kalman, structural decomposition, robust optimization, and Bayesian model averaging were all considered and set aside — see rationale below) for being immediately buildable from data already on disk without imposing assumptions the data can't support.

**1. Two uncertainty-aware alternatives to the pooled point estimate** (`scripts/build_eta_uncertainty_distribution.py`, `data/processed/eta_uncertainty.json`):
- **Non-parametric bootstrap**: resamples each tier's 6–7 real per-cycle point estimates with replacement — no assumption about shape, which matters since the real data include a sign flip (Toss-Up: $-0.22$ in 2016) a fitted Normal would smooth away.
- **Random-effects meta-analysis (DerSimonian-Laird)**: separates real between-cycle variance ($\tau^2$) from each cycle's own estimation noise, reporting $I^2$ (share of cross-cycle spread that's real, not noise) and a predictive SD for an unseen cycle. $I^2$ independently reproduces §5's significance results: 0% for Lean R (matches its non-significant joint test), 70–89% for the five significant tiers.
- **A genuinely new finding surfaced by building this**: the properly cycle-weighted pooled means differ substantially from the naive pooled-panel numbers already in Paper III §4.4 — e.g. Lean D drops from 0.637 to 0.277, Safe R from 0.584 to 0.258 — because naive pooling lets whichever cycle generated the most transaction rows dominate, while cycle-weighting gives each *election* a vote scaled by its own precision. Paper III §4.4 now carries both figures with the cycle-weighted one flagged as the better estimate.

**2. Wiring a per-path draw into the Monte Carlo** (`scripts/solve_bellman_lsm.py`): `run_lsm()` was refactored to take `eta_arr_by_path`/`resid_std_arr_by_path` of shape `(K_PATHS, n)` instead of a single dict broadcast to every path, unifying the old two-bracket behavior (`tile_single_cycle()`, identical value tiled across all paths) and the new bootstrap behavior (`bootstrap_eta_resid_paths()`, each path draws its own per-tier $(\hat\eta, \text{resid\_std})$ pair from one randomly chosen historical cycle, held fixed for that whole path). A new `eta_bootstrap_all_cycles` scenario runs alongside the original two brackets, not replacing them. Result: $\Theta(0)=-0.414$, and `frac_deploy_now=91.5\%$ — the first *non-degenerate* $t=0$ statistic this project has produced, since paths under bootstrap genuinely differ from each other even before any period elapses (unlike the single-cycle brackets, where every path shares an identical pre-shock state and the 0%/100% split is closer to an artifact of that than a real distribution). Full detail in Paper III §8.2. *(Superseded by §9, added 2026-07-19: after the `beta1_open` correction, this scenario's $t=0$ split closes entirely — $\Theta(0)=-0.289$, `frac_deploy_now=100.0\%$. The 91.5% figure was real at the time and is kept here as the historical record of the first non-degenerate split this project produced, not retracted — it just no longer describes the corrected model.)*

**Why the other six brainstormed options were set aside** (not because they're wrong, but because $n=5$–$7$ cycles per tier can't support them yet):
- *Hierarchical Bayesian (tier × cycle crossed random effects)* — the right long-run answer, and DerSimonian-Laird above is a single-tier special case of it; a full crossed model is more work for a modest gain at this sample size and is a natural next step, not implemented here.
- *Hidden regimes* — only identifiable from an externally pre-specified rule (midterm/presidential, a recognized wave-election definition), not from the eta values themselves (which would be circular); not attempted.
- *State-space/Kalman random walk* — hits the same wall the OU-drift fit already hit with far more data (§0.1: $p=0.37$ pooling thousands of points); with 5–7 points per tier, a transition variance isn't identifiable. The data itself argues against this framing too: Toss-Up's sequence ($+0.42,+0.25,-0.22,+0.08,+0.11,+0.73,+0.34$) bounces and snaps back rather than wandering and staying wandered, more consistent with noise-around-a-mean than a persistent random walk.
- *Structural decomposition* ($\eta=\eta_{\text{fundraising}}+\eta_{\text{media}}+\eta_{\text{polarization}}$) — no data in this repository attributes IE spending to a causal channel; a different research program, not an extension of this one.
- *Robust optimization* (max-min over $\eta\in[\eta_{\min},\eta_{\max}]$) — a genuinely different, assumption-light paradigm worth prototyping separately; not implemented here since it changes the optimizer's objective, not just the calibration inputs.
- *Bayesian model averaging* (pooled vs. 2022 vs. 2024) — redundant with the bootstrap/random-effects approach above: these were never really competing model specifications, just different subsets of the same data.

## 7. A 1-year-horizon counterfactual, and what it reveals about the model's structure (added 2026-07-17; re-run 2026-07-20 against §9's `beta1_open` correction)

*Note added 2026-07-20: this section has now been re-run (`scripts/solve_bellman_lsm_1yr_counterfactual.py`, unmodified, against the corrected `run_lsm()`) and the numbers below are current. The pre-correction snapshot is preserved at `outputs/pre_beta1_open_fix/theta_schedule_1yr_counterfactual.json` for the record. One consequence worth flagging up front: since §9's correction closed `eta_bootstrap_all_cycles`'s live-horizon split to unanimous 100% (it was 91.5% when this section was first written), the original "reversal" framing below — a scenario with real wait mass flipping to full deployment at the longer horizon — no longer has a live data point to reverse *from*. The mechanism is unchanged and still verifiable, just visible now only in $\Theta(0)$'s magnitude getting more negative at the longer horizon, not in a fraction-of-paths flip.*

§6's `eta_bootstrap_all_cycles` scenario was the first to show a non-degenerate $t=0$ split (`frac_deploy_now=91.5\%`, since superseded by §9). A colleague reviewing that result asked the natural question: is 91.5% (rather than 100%) mostly a function of the live decision sitting only ~98 days from Election Day — i.e., would deploy-now fall substantially if the same model were run from a full year out, since more remaining time should mean more option value in waiting? A controlled counterfactual was built to test this directly (`scripts/solve_bellman_lsm_1yr_counterfactual.py`), and it did not confirm the prediction — at the time, it reversed outright (see below); after §9's correction, the live scenario no longer has wait mass to reverse, but the same underlying mechanism still strengthens deploy-now at the longer horizon in every scenario, now measured in $\Theta(0)$'s magnitude rather than in `frac_deploy_now`.

**Design.** The script monkey-patches `solve_bellman_lsm`'s module-level `TODAY`/`N_PERIODS` globals (`ELECTION_DAY - 364 days`, giving 26 biweekly periods instead of the live 7) before calling the same `build_universe()`, `run_lsm()`, `tile_single_cycle()`, and `bootstrap_eta_resid_paths()` functions, unmodified. **Explicitly not** a reconstruction of what the 2026 race universe looked like a year ago — §0.1.1 already established no dated candidate-committee floor panel exists to build that from. Instead, today's actual observed starting state (floors, $R_{i,t}$, $\sigma_i$) is held exactly fixed, and only the artificial "days until Election Day" parameter changes — isolating the pure time-horizon effect, at the cost of not being a literal history. Both this run's $t=0$ and the live run's $t=0$ are true, dispersion-free starting points (no epsilon has accumulated yet in either), so comparing $t=0$-to-$t=0$ across the two runs is a clean test; comparing any other matched days-remaining point between them is not, since `remaining_variance()` is anchored to absolute days-remaining and the 364-day run has already burned through much of its variance budget by the time it reaches, say, 98 days remaining mid-path.

**Results** (`outputs/theta_schedule_1yr_counterfactual.json`, current as of the §9 correction):

| | Live, 98 days out | 1-year counterfactual, 364 days out | (pre-§9 counterfactual, for reference) |
|---|---|---|---|
| eta_fit_2022 | $\Theta(0)=-0.633$, deploy=100.0% | $\Theta(0)=-0.746$, deploy=**100.0%** | $-0.628$, 100.0% |
| eta_fit_2024 | $\Theta(0)=-0.109$, deploy=100.0% | $\Theta(0)=-0.174$, deploy=**100.0%** | $-0.122$, 100.0% |
| eta_bootstrap_all_cycles | $\Theta(0)=-0.289$, deploy=**100.0%** | $\Theta(0)=-0.447$, deploy=**100.0%** | $-0.586$, 100.0% |

**The magnitude effect this section originally found is intact and, if anything, slightly larger in two of three scenarios: $\Theta(0)$ gets more negative at the 364-day horizon than at the live 98-day horizon, in every scenario, meaning deploy-now is *more* strongly favored the further out the decision is made, not less.** This is the same direction as the original finding. What has changed is that `eta_bootstrap_all_cycles` no longer supplies the dramatic "8.5% wait mass at the live horizon flips to 0% at the long horizon" data point, because §9's correction already closed that wait mass at the live horizon on its own — there is nothing left for the longer horizon to flip. The underlying mechanism (below) does not depend on that particular scenario having live-horizon wait mass to demonstrate; it is visible directly in $\Theta(0)$'s magnitude in all three scenarios and is a real, deterministic feature of the model, not noise (the Bellman recursion has no randomness in which branch wins conditional on the inputs already drawn), so it was traced to its mechanical source rather than left as "we don't know why."

**Mechanism, verified directly from the code, not inferred:**

- **Discounting is not the cause — there isn't any.** `V_star = np.where(deploy_now, deploy_vals, wait_vals)` (`solve_bellman_lsm.py:355`) carries no $\beta^t$ term anywhere in the recursion; the only `gamma` in the file is the LP allocator's risk-aversion coefficient (`allocator.py`), unrelated to time. There is no discount factor to vary between the two runs.
- **Deploying is not a persistent state improvement that compounds over more future periods — it's a one-shot absorbing action.** `_deploy_value()` sets `d_t = floor_arr.copy()` (`solve_bellman_lsm.py:299`) at every backward-induction step, always resetting to the *original* candidate floor. A path's earlier deploy decision never updates the state later periods see. This is optimal-stopping (American-option) structure, not a control problem where an early action keeps paying off.
- **The actual mechanism is the deploy branch's variance-convolution identity, and it is quantitative, not speculative.** The deploy branch evaluates $\Phi(\mu_i/\sqrt{\sigma_i^2+V(t)})$ rather than $\Phi(\mu_i/\sigma_i)$ — a closed-form application of $\mathbb E[\Phi((\mu+\xi)/\sigma)]=\Phi(\mu/\sqrt{\sigma^2+V})$ for $\xi\sim N(0,V)$ — specifically so that deploying now already analytically integrates over the effect of *all* remaining idiosyncratic drift between now and Election Day, in one step, rather than requiring the model to actually wait through it. $V(t)=\sigma^2(1-e^{-\lambda \cdot \text{days\_remaining}})$, using the fitted $\lambda=0.00536$/day, gives $V(t)/\sigma^2 = 41\%$ at 98 days remaining but $86\%$ at 364 days remaining. As the horizon grows, deploying-now's built-in approximation to "wait and see" captures a larger and larger share of the uncertainty that literal waiting would resolve — shrinking the residual, distinct value that actual sequential waiting still has left to add. That is what falls as the horizon lengthens, not the value of information itself.

**Sharper statement of the finding, and its limit:** the apparent reduction in option value with longer horizons is not a generic property of optimal stopping — it arises because the deploy branch analytically integrates over a growing fraction of future idiosyncratic uncertainty through the variance-convolution identity. As the remaining horizon increases, this approximation captures an increasing share of the uncertainty that explicit waiting would otherwise resolve (41% at 98 days vs. 86% at 364 days, under the fitted decay rate), reducing the incremental value of sequential observation. Taken to its limit, as $\text{days\_remaining}\to\infty$, $V(t)\to\sigma^2$ and the deploy branch integrates over almost the full uncertainty budget without giving up the ability to act today — a very favorable approximation for "deploy now" to hold. The convolution identity is mathematically correct for what it integrates over: the expected effect of unresolved Gaussian idiosyncratic shocks on the terminal win probability. **What it does not, and structurally cannot, capture is the value of adaptive decision-making** — i.e., that new information arriving mid-campaign could change *how* the reserve is allocated, not just *whether* it's deployed. In this model, deployment is a one-time absorbing decision and the allocation is computed once, at the moment of deployment (§0.1.1's data gap is exactly why the wait branch can't even let $\eta$ react to a non-existent spending trickle); that adaptive channel is intentionally absent from the model as built, which is precisely why the convolution can substitute for so much of waiting's value. Put plainly: this model solves an optimal-stopping problem with analytical expectation-substitution for future shocks, not a fully adaptive sequential control problem — and once that distinction is visible, the 1-year counterfactual's direction stops being surprising and becomes an expected consequence of the model's own construction. This is worth stating explicitly in Paper III as a scope boundary (alongside §0.1.1's other absorbing-decision caveat), not left implicit: any reader tempted to interpret $\Theta(t)$ as "the value of flexibility" in the fully general sense should read it instead as "the value of flexibility *net of* what the deploy branch's own analytical widening already captures for free" — a narrower and more accurate claim.

## 8. The continuous-deployment-fraction generalization, implemented and run: corners persist (added 2026-07-17)

§1.3's design is now implemented (`scripts/solve_bellman_lsm_continuous_phi.py`) and run at $K=2000$ across all three $\eta$ scenarios (`eta_fit_2022`, `eta_fit_2024`, `eta_bootstrap_all_cycles`), 5-point grid ($\mathcal F=\{0,0.25,0.5,0.75,1.0\}\times F_0$), then spot-checked at an 11-point grid (10%-step) per §1.3 item 6's sensitivity requirement. **The answer is no: allowing a genuinely continuous, sequential deployment fraction does not surface an interior optimum. Every scenario recommends full deployment today, unanimously across all simulated paths, at both grid resolutions.**

### 8.1 Implementation notes and two deliberate departures from §1.3's literal text

The floor-reset bug (§1.3 item 2: `_deploy_value()` hardcoding `d_t = floor_arr.copy()` regardless of how much has already been committed) is fixed by construction: `_solve_committed_floor()` computes the per-race allocation for a *target cumulative budget* $C_g$ via the same LP call `_deploy_value()` already makes, but genuinely parameterized by $C_g$ instead of frozen at $F_0$. This is precomputed once per (grid level, path, period) — $4\times2000\times8=64{,}000$ LP calls at 5 points, $10\times2000\times8=160{,}000$ at 11 points — rather than once per (starting-state, action) pair as §1.3 item 5's literal accounting assumed (which would have given $\sim210{,}000$ and $\sim170$-minute figures); collapsing the redundant recomputation this way is a genuine efficiency gain, not just a simplification, and kept both the 5-point ($K=2000$, ~12 min/scenario) and 11-point ($K=500$ spot-check, ~8 min) runs tractable in this session.

Two departures from §1.3's literal text, both made deliberately and both load-bearing for the result:

1. **Exact nonlinear valuation instead of a linearized chain-rule delta.** §1.3's original phrasing (written before implementation) suggested extending `_deploy_value()`'s existing pattern of computing a linearized `delta_mu = grad * delta_s` and adding it to `mu_t`. Implemented instead: once `_solve_committed_floor()` returns the resulting per-race floor, this script recomputes $\mu$ directly from the exact structural formula already used everywhere else in `solve_bellman_lsm.py` ($\alpha_0+\alpha_1\text{pvi}+\alpha_2\mathbb 1[\text{incumbent}]+\alpha_3 GB+c\cdot\log(D/(D+R))$), with $R$ increased by $\eta_i\cdot(D_{\text{new}}-D_{\text{floor}})$ — i.e. paper3_draft.md §4.1's own $R_i(D_i)=R_{\text{base}}+\eta\cdot\max(0,\text{party}_i-\text{party}_{i,\text{obs}})$ applied directly, rather than folded into a chain-rule slope. This is strictly more faithful to §4.1's stated specification than a linearized delta would have been, and it is what gives the value-of-budget curve genuine, non-approximated concavity ($\log(D/(D+R))$ is itself concave in $D$) — avoiding exactly the class of frozen-linearization bug this whole exercise exists to fix. The LP allocator's *choice* of which races receive money still uses the linearized, $\eta$-discounted MSG objective (§0.1.2), since the fast LP's objective must be linear — but valuing the resulting allocation does not have to inherit that approximation, and here it doesn't.
2. **Grid-stratified regression instead of the unified quadratic-in-remaining-budget basis §1.3 item 4 stated as preferred.** Discovered empirically while smoke-testing at small $K$: pooling all grid states into one regression with $f'$/$(f')^2$ as extra features produced visibly non-monotonic continuation-value predictions across budget levels — the fitted quadratic was a poor functional form for what turned out to be a genuinely *step-shaped* value-of-budget relationship (see §8.2), not the smooth curve the unified basis assumes. Switching to a separate, unconstrained OLS fit per grid state (no shape imposed across budget levels) fixed this immediately. §1.3's stated reason to prefer unified — sparse cells under endogenous action-sampling — does not apply to this implementation's design anyway: every grid state is evaluated densely for all $K$ paths at every period (mu_committed is precomputed for the full grid $\times$ full path set, not sampled along a single forward-simulated policy trajectory), so there is no sparse-cell risk to trade off against.

### 8.2 Why the value-of-budget curve is a step, not a smooth concave curve

The isolated mechanism (checked directly, independent of the backward induction; not re-run against §9's `beta1_open` correction — the absolute levels below would shift by roughly the same ~0.4–0.6 seats documented in §9, but the step *shape*, driven by the LP allocator's knapsack degeneracy rather than by the mu-level formula, is not expected to change): at $t=0$'s baseline state, $E[\text{Seats}]$ rises from 221.65 (nothing committed) to 225.64 at just 5% of $F_0$ committed, to 225.78 at 25%, to 225.81 at 100% — a huge jump in the first few percent of the budget, then near-total flatness. This traces directly to a mechanism already documented in paper3_draft.md §8.2's addendum: the LP allocator's objective (`maximize msg @ s`) is a pure greedy knapsack with no diminishing-returns mechanism within a call, so it funds the **same fixed set of $\sim$7 low-floor outlier races regardless of total budget** — larger budgets just mean those same 7 races receive proportionally more money each, which saturates $\log(D/(D+R))$ (since $D$ quickly dominates $R$ for these historically under-funded races) almost immediately. The 11-point grid confirms this at finer resolution directly: option values for 10% through 90% committed sit within 0.001 seats of each other (226.2531–226.2544), while both corners (hold, 226.3625; deploy-all, 226.4702) sit clearly above the entire interior range.

### 8.4 Interpretation: §1.1's motivating hypothesis is not confirmed

§1.1 asked whether the binary framing was masking a genuine interior reserve — plausible on its face, since `eta_fit_2024`'s $\Theta(0)=-0.039$ sat an order of magnitude closer to indifference than `eta_fit_2022`'s $-0.517$ (post-§9-correction: $-0.109$ vs. $-0.633$ — still the closer-to-indifference bracket by roughly the same relative margin). Having now built the properly concave, sequential, carried-forward-budget version, **the answer is no**, and the reason is economically legible rather than a modeling accident: any partial commitment pays the full cost of foreclosing flexibility on the committed portion (money once given to the same 7 knapsack-selected races cannot be un-given, and doesn't buy access to a different, better opportunity later) without capturing the deploy branch's exclusive benefit — the variance-convolution identity (§7 above; paper3_draft.md §8.3) that analytically integrates over *all* remaining idiosyncratic drift in one step. Holding entirely preserves 100% of that flexibility; deploying fully captures 100% of the analytic-integration benefit; anything in between gets a diluted share of the first with none of the second's completeness. Given the LP's own knapsack degeneracy also means there is essentially nothing left to gain past the first few percent of the budget, a genuine interior optimum was never structurally available to find — not because the binary framing failed to express it, but because this model's actual economics don't produce one. This should be read together with, not as a contradiction of, §7's scope-boundary finding: $\Theta(t)$ here measures the value of flexibility *net of* the deploy branch's own analytical widening, and that same widening is precisely why fractional deployment can't compete with either corner.

## 9. A third correction: `beta1_open` was never substituted into `mu_struct`, only into the gradient (found and fixed 2026-07-19)

**How this was found.** Not from a data audit or a modeling question — from writing the first automated test coverage this project's Bellman/LSM machinery has ever had (`tests/test_bellman_lsm.py`; before this pass, `scripts/solve_bellman_lsm.py`, `scripts/solve_bellman_lsm_continuous_phi.py`, and `scripts/simulate_and_validate.py` — 1,373 lines, including where both prior corrections in this document lived — had zero test coverage). Writing a cross-consistency test between `solve_bellman_lsm.py`'s inline `mu_struct` computation and `solve_bellman_lsm_continuous_phi.py`'s `_mu_struct()` (checking a docstring's claim that the two are "identical" against the actual code, not just trusting the comment) surfaced that both correctly reproduce each other — but a further check against `margin_gradient()` and the static pipeline's `win_prob.predict()` found that `margin_gradient()` already substitutes `coef.beta1_open` for Open-seat races (has done since §0.1.2 of this document), while both copies of `mu_struct` unconditionally used `coef.beta1`, even for Open seats. `beta1_open=6.977` vs. `beta1=5.457` (data_catalog.md §3.1) is not a small gap.

**Why this is a real bug, not a stylistic inconsistency.** The Bellman/Theta machinery computed the *level* of $\mu_i$ for every Open-seat race with one spending elasticity and the *gradient* of that same $\mu_i$ — which the LP allocator uses to decide where money goes — with a different one, in the same backward-induction step. The live 2026 universe has 49 Open-seat races (11% of the 434-race universe), so this was not a corner case.

**The fix.** `run_lsm()` (`scripts/solve_bellman_lsm.py`) now computes a per-race `beta1_eff_arr`, substituting `beta1_open` for Open-seat races before building `mu_struct`, exactly matching `margin_gradient()`'s existing branch. `_mu_struct()` (`scripts/solve_bellman_lsm_continuous_phi.py`) gained an `is_open_arr` parameter doing the same substitution; `_setup_universe()` and both call sites were updated to thread it through. Both changes are covered by direct tests (`tests/test_bellman_lsm.py::TestMuStructConsistency`), including one confirming the fixed formula matches `win_prob.predict()` — the static pipeline's already-correct source of truth for what `beta1_open` should do — for every Open-seat race in a synthetic universe.

## 10. Sanity-checking the 1-year counterfactual against domain intuition: does the data support a spending-timing-decay explanation? (added 2026-07-20)

**Motivation.** §7's re-confirmed finding — $\Theta(0)$ gets *more* negative (deploy-now *more* strongly favored) at a 364-day horizon than at the live 98-day horizon, in every scenario — should not survive a basic smell test unexamined: no real campaign committee would treat "deploy the entire reserve a year before Election Day" as good practice, and a model that recommends it more confidently than deploying today is a model worth doubting, not defending. Two structural candidates were already on record for why this could happen even though the recursion is computed correctly (§7's own scope-boundary language): (a) the model cannot express *adaptive reallocation* — deployment is a single one-shot LP solve, so "waiting" only ever means "do nothing, then eventually deploy once," never "watch which races heat up and target the reserve accordingly"; and (b) the underlying margin model was fit on **cycle-cumulative** $D_{\text{total}}$/$R_{\text{total}}$ (`fec.py::build_total_spend()` sums candidate + coordinated + IE spend over the *entire* cycle, with no filing-date granularity for two of the three components), so it has no way to represent a dollar spent in February differently from a dollar spent in October — if real spending's persuasive effect decays over time, the model simply cannot know that. (b) was not previously tested against data; this section tests it.

### 10.1 Design, and a known data constraint stated up front

Two checks (`scripts/test_spending_timing_decay.py`), both on the 2022/2024 held-out cycles already used for Section 5.4/5.5's Validation A, restricted to the same competitive-tier subset:

- **Test 1 (correlation strength across the cycle).** Extends Validation A's single September-1 snapshot to nine snapshots spread from 270 to 30 days before Election Day (`dynamic/simulate.py::_reconstruct_races_at`, the same point-in-time reconstruction machinery Validation A already uses), Spearman-correlating each snapshot's $\mu_i$ against realized November margin. If early-cycle financial state predicts the outcome about as well as late-cycle state, that argues against a strong timing effect; if early snapshots correlate much more weakly, that's *consistent with* (but not dispositive of) decay.
- **Test 2 (front-loading residual test, the sharper one).** For each competitive race with meaningful independent-expenditure activity on both sides, computes each party's IE-spend share that occurred before September 1 relative to its full-cycle total, and tests whether a race's *asymmetry* in front-loading (D's early share minus R's early share) predicts a systematic residual after controlling for the standard, unmodified, final-ratio prediction (`build_universe()`'s cycle-cumulative totals — exactly what Paper I's model already uses everywhere else). A negative relationship (D front-loading relative to R predicts D *underperforming* the final-ratio prediction) would be direct evidence that early money "counts for less" than the static model assumes.

**Known constraint, stated before reporting results, not after:** `_static_floor_totals()`'s own docstring already documents that candidate-committee and coordinated-expenditure spend have **no per-filing-date source anywhere in this repository** — only independent-expenditure spend is genuinely date-bucketed (`fec.cumulative_ie_as_of`). Every "point-in-time" snapshot below therefore holds candidate+coordinated spend at its full, *final* cycle value at every date and only varies the IE component. This is the same constraint Validation A already operates under, not a new one introduced here, but it means both tests are checks of **IE-spend timing specifically**, not all-source spend timing — a partial test of the hypothesis, reported as such.

### 10.2 Results: both formal tests are null — and the reason why is more informative than either test's own result

**Test 1 is flat, but for an uninformative reason.** Spearman $\rho$ is *bit-for-bit identical* across five consecutive snapshot dates (270 through 120 days out) in both cycles — 0.4555 throughout 2022, 0.6278 throughout 2024 — before nudging up slightly in the final ~60 days (0.471 at 60d, 0.469 at 30d in 2022; 0.628 at 60d, 0.666 at 30d in 2024). This is not evidence that early money predicts as well as late money; it is evidence that almost nothing had moved yet at any of the "early" snapshots to distinguish them from each other.

**Test 2 came back null** (Spearman $\rho$(frontload asymmetry, residual) $=+0.015$, $p=0.91$, $n=51$ in 2022; $\rho=-0.048$, $p=0.76$, $n=42$ in 2024 — OLS slopes similarly indistinguishable from zero in both cycles) — but checking *why* the test had so little power surfaced the actual finding:

**By September 1, Democratic-aligned IE spend averaged just 2.2% (2022) / 1.6% (2024) of its eventual full-cycle total in competitive races; Republican-aligned averaged 3.7% / 4.4%.** There is essentially no genuinely front-loaded comparison group anywhere in the real data — on average, 95–98% of all IE money in these races was spent in the final two months of the cycle, every cycle, both parties. Test 1's flatness and Test 2's null result are both direct consequences of this: there simply is not enough real-world timing variation in the one channel this repo can date-bucket to detect a decay effect from, in either direction.

### 10.3 Interpretation: the formal tests are inconclusive, but the underlying fact they surfaced is not

Neither test confirms nor rules out spending-effect decay specifically — they were underpowered by the very data they were built from, and that is reported honestly rather than stretched into a false negative ("no decay detected" would overstate what a test with almost no early-spend variation to work with can actually show). But the fact responsible for that lack of power is itself a stronger, more direct piece of evidence than either formal test: **real independent-expenditure committees — sophisticated, professional political spenders whose entire function is deploying money efficiently to win elections — do not spend anything like what this model's 1-year counterfactual recommends.** They hold back 95%+ of their money until the final ~60 days, consistently, across two different cycles and both parties. That revealed behavior directly contradicts "deploy the full reserve with a year still to go, more confidently than deploying today," for reasons this project's model has no channel to represent, whether the true reason is persuasion decay, targeting information not being good enough yet a year out (i.e. exactly §7's already-documented adaptive-reallocation gap), strategic waiting to see how the opponent moves first, or some combination.

**This does not overturn §7's finding as a correct consequence of the model as specified** — the backward induction is doing exactly what a model with no decay mechanism and no adaptive-reallocation channel should do, and reporting that honestly (§7, §8.5) remains the right call. What this section adds is a concrete, data-grounded reason **not to read the 1-year-counterfactual number as real-world guidance**: whatever the true mechanism, actual political money overwhelmingly disagrees with the model's long-horizon recommendation, and this project currently has no calibrated channel — decay, adaptive targeting, or otherwise — that could let the model agree with observed practitioner behavior even if the underlying $P$ were otherwise perfectly specified. Building such a channel (most plausibly, letting the wait branch's continuation value reflect a *genuine* future re-targeting option rather than the current one-shot deployment) is a substantial extension, not a quick fix, and is not attempted here — but it is now the best-evidenced concrete next step for this specific finding, better motivated than either of §7's original ruled-out mechanisms (discounting, persistent state improvement) ever were. `outputs/spending_timing_decay_check.json` is the saved result.

## 11. A necessary condition for dynamic optimization: does the optimal allocation actually change with information? (added 2026-07-20)

Before asking whether *preserving* the ability to reallocate has positive expected value (§§1, 7, 8 — the whole $\Theta$ apparatus), a logically prior question has to be settled first, and had not been checked directly anywhere in this research line until now: **does new information even change what the optimizer wants to do?** If the answer were no — if the optimal allocation barely moved as idiosyncratic uncertainty resolved — then building an adaptive-reallocation channel into the Bellman machinery (§10's proposed next step) would be solving a problem that doesn't exist, regardless of how the timing-decay or Θ findings came out. This had never been tested in isolation from the rest of the machinery; it is now.

**Design** (`scripts/test_snapshot_targeting_shift.py`). Re-solve `optimize_nonlinear()` — the allocator that actually respects diminishing returns, not the LP's knapsack-degenerate `optimize()` (paper3_draft.md §8.2's addendum) — using real historical state snapshots at two points in each of the 2022 and 2024 held-out cycles (60 days and 14 days before Election Day, chosen using §10's own finding about where realized outcomes start to differentiate from the flat 270-to-90-day window), **holding the total real, final-cycle DCCC party budget fixed and identical at both snapshots**. This isolates the question cleanly: does information change *targeting*, independent of how much money there is to spend? Candidate-committee floors are frozen at their true final value at both snapshots regardless (the same data constraint as §10); only the opponent's dated IE spend genuinely varies the ratio each snapshot sees.

**Results:**

| Cycle | Jaccard overlap, top 20 | Races only in 60d-out top 20 | Races only in 14d-out top 20 | Spearman $\rho$, full allocation vector |
|---|---|---|---|---|
| 2022 | **0.54** | AZ-01, CA-45, FL-28, NJ-02, TX-23, WI-01 | CA-22, ME-02, NH-01, OH-01, PA-07, PA-08 | 0.698 |
| 2024 | **0.67** | FL-15, IA-03, PA-01, TX-23 | CA-13, CA-22, PA-08, WA-03 | 0.651 |

Roughly a third to nearly half of the top-20 targeted races turn over between the two snapshots, in both cycles, using the *same* total dollars each time. This is not a marginal or ignorable reshuffling — it is the scale of turnover that would visibly change a real DCCC targeting memo.

**Framing this section is written around** (a colleague's framing, adopted directly because it states the logical structure better than an after-the-fact description would): *A necessary condition for dynamic optimization is that the optimal allocation changes as information arrives. We test this directly by re-solving the nonlinear optimizer using historical state snapshots while holding the total budget fixed. Across both 2022 and 2024, approximately one-third to one-half of the top twenty targeted races change, with Jaccard overlaps of 0.54 and 0.67, respectively. Thus, even before considering the value of sequential decision-making, the allocation policy itself is demonstrably state-dependent. This establishes that adaptive reallocation is a meaningful decision problem rather than a theoretical abstraction.* Having shown that the optimal policy evolves with information, the natural next question — the one $\Theta$ actually answers — is whether *preserving the ability to adapt* has positive expected value.

**This separates two questions this research line had previously left conflated, and both findings survive independently:**

1. **Does new information materially change what the optimizer wants to do?** Yes — this section's historical snapshot test shows substantial turnover in optimal targeting, holding budget fixed, in both real cycles tested.
2. **Is the expected value of waiting large enough to justify delaying deployment?** No — under the current, honestly-scoped state-transition model $P$, the Bellman analysis (§§0, 4, 6, 7, 8, 9) finds that immediate deployment maximizes expected seats in every scenario tested, at both grid resolutions.

These are complementary, not contradictory. (1) justifies modeling this as a dynamic problem at all — it rules out the possibility that the whole research line was answering a question nobody needed to ask. (2) characterizes the optimal policy *within the assumptions of the current $P$* — and, per §10, those assumptions are missing a real channel (most plausibly: letting the wait branch's continuation value reflect genuine future re-targeting, rather than the current one-shot deployment) that this section's own result shows would have real, exploitable content if built. The gap between (1) and (2) is exactly what a correctly-scoped adaptive-reallocation extension would close — not a contradiction to be explained away, but the honest current boundary of $P$.

One caveat, stated plainly rather than glossed over: this test measures whether *targets* move, not how much *expected-seats value* comes from being able to move them — that requires an actual sequential value comparison (a real "commit at 60d" vs. "wait and commit at 14d" Bellman-style comparison), not two static snapshots, and is not what this section does. The E[Seats] levels reported per snapshot (224.85 → 223.00 in 2022; 221.84 → 221.43 in 2024) should not be read as "waiting made the outcome worse" — the decline reflects the opponent's own late spending being more fully captured by the later snapshot, not the committee's own decision quality. `outputs/snapshot_targeting_shift_check.json` is the saved result.

## 12. Two documented "permanent" data gaps turn out not to be — and the fix they unblock (added 2026-07-20)

Responding to external reviewer feedback on Paper III's missing-data catalog, this section re-checked two gaps this document had previously reported as effectively closed off: §0.1.1's "no per-filing date field anywhere in this repository" for candidate-committee spending, and §6.2's "district polling too sparse to fit a real process." The first turns out to be wrong — checked directly against the live FEC API, not re-derived from documentation — and the second is confirmed correct for historical estimation but reveals a real, previously-unused *live* signal.

### 12.1 What was actually wrong, and how it was found

§0.1.1's claim was scoped correctly to what this project had checked: `candidate_disbursements_{cycle}.csv` (the bulk `weball` file this project's fetch pipeline reads) genuinely has no per-filing-date field — that part is still true. What was never checked is whether the FEC API exposes a *different* endpoint with dated data. It does: `/v1/committee/{committee_id}/reports/` returns every periodic Form 3 filing (quarterly, pre-general, post-general, year-end) a committee has ever made, each with its own `coverage_start_date`, `coverage_end_date`, `total_receipts_period`, `total_disbursements_period`, and `cash_on_hand_end_period` — confirmed live against both a party committee (DCCC) and a real House candidate committee, returning real dated 2024 and 2026 data, before any code was written.

Separately, VoteHub's polling API — already used by `scripts/fetch_polling.py` for generic-ballot polls — turns out to carry a `poll_type=us-representative` category with real per-district subjects (`"2026 PA-07"`, etc.), not just national polls. Checked directly: 28 districts have any live 2026 coverage at all, and critically, **no historical (2022/2024) district-level data exists at this endpoint** — so this does not reopen §6.2's finding that a historical $\varepsilon_{i,t}$ process cannot currently be estimated from data; it adds a new *live* diagnostic signal only.

### 12.2 What was built

1. **`scripts/fetch_data.py::fetch_candidate_periodic_reports()`** + **`src/backtest/data/fec.py`**'s dated-candidate loaders (`load_candidate_periodic_reports`, `cumulative_candidate_spend_as_of`, `cumulative_candidate_receipts_as_of`, `cash_on_hand_as_of`, `spend_velocity`, `receipts_velocity`) — the same dated-panel pattern already established for independent expenditures (`load_ie_transactions_dated`/`cumulative_ie_as_of`), including amendment resolution (kept the highest `beginning_image_number` per `(committee_id, coverage_start_date, coverage_end_date)`, an approximation stated as such, not an exact `file_num`/`prev_file_num` chain like the IE handling). Full schema in `data_catalog.md` §2.7.
2. **`dynamic/simulate.py`'s historical reconstruction** (`_reconstruct_races_at`) now uses genuinely dated candidate-committee spend instead of holding it fixed at the cycle-final total — falling back cleanly to the old fixed-total behavior, with a logged warning, for any cycle the dated panel hasn't been fetched for yet (mirrors this project's established honest-fallback discipline, e.g. `AdReservationProxySource`). `dynamic/ledger.py`'s `RealizedSpendCommitmentSource` docstring is updated to reflect the narrowed remaining gap (coordinated-expenditure dating only — Schedule F, unexplored, not solved by this pass).
3. **§0.1.1's actual fix**: `scripts/estimate_candidate_spend_trickle.py` fits a per-tier candidate-committee disbursement growth rate ($/day, median preferred over mean given lumpy FEC filing timing) from the dated panel. `scripts/solve_bellman_lsm.py`'s wait branch now grows $D_{i,t}$ at this calibrated rate instead of holding it perfectly fixed, and $R_{i,t}$ reacts to that growth via the already-estimated $\hat\eta(\text{tier})$ — exactly the mechanism this section originally proposed and could not build. A related bug fixed in the same pass: `_deploy_value()` previously reset to the period-0 candidate floor (`floor_arr.copy()`) at every backward-induction step regardless of `tstep`; it now receives the current (trickled) floor for that period/path, so a deploy decision made later in the campaign correctly adds discretionary spend on top of however much the candidate's own committee has raised and spent by then, not the day-0 baseline.
4. **A stated scope boundary, not silently assumed away**: $\hat\eta(\text{tier})$ was estimated from IE-to-IE reaction only (§4.1's stated scope). Applying it here to $R_{i,t}$'s reaction to a candidate-committee spending trickle (rather than an IE increment) is a new, untested application of an existing estimate — the same kind of assumption §5.5 already makes explicit for $\alpha_3$. `simulate_and_validate.py`'s §7.1 self-consistency gate still checks $\hat\eta$ recovery against the IE-to-IE mechanism it was fit on (Check B), not this new candidate-spend-to-IE channel, which would need its own regression to validate directly — not attempted in this pass, reported as an open item rather than assumed validated.
5. **Live district polling**: `scripts/fetch_polling.py::fetch_house_district_polls()` + `house_district_poll_summary()`, and four new stub fields on `dynamic/state.py`'s `RaceState` (`poll_mean_t`, `poll_sigma_t`, `poll_n_t`, `poll_trend_t`) — live-2026-only, never wired into $\mu_i$, same treatment as the pre-existing generic-ballot-polls precedent, per §12.1's finding that no historical panel exists to validate a structural coefficient against.
6. `dynamic/updates.py::compute_raw_snapshot()` gained an optional `cash_on_hand_by_district` parameter, finally giving `RaceState.cash_on_hand_d` (a stub since Paper II) a real data source; wired through `dynamic/simulate.py`'s historical harness whenever the dated panel is available for that cycle.

All of the above is covered by new tests (`tests/test_fec.py`, `tests/test_fetch_polling.py`, plus extensions to `tests/test_dynamic_simulate.py`, `tests/test_dynamic_updates.py`, and `tests/test_bellman_lsm.py`'s new `TestSpendTrickle` class, which confirms the trickle mechanism is actually wired into `run_lsm()`'s output, not just present-but-inert — the same class of bug (`eta` computed but never multiplied into anything) §0 of this document already found once before).

### 12.3 The production re-run, completed — and a fourth correction found before trusting it

The fetch completed (2022 and 2024 — the same two cycles $\hat\eta$ was originally calibrated on; §12.5 discusses extending to the full 2012–2024 panel as future work), scoped to `fec.load_candidate_disbursements()`'s nominee roster (~875/872 candidates per cycle) rather than every primary candidate in the raw bulk file (~3,300+ per cycle) — the latter would have silently double-counted district spend by summing primary losers' committees together with the actual nominee's, a real correctness bug caught before it ever produced a number (see `fetch_candidate_periodic_reports`'s docstring for the full reasoning). The fetch itself needed several checkpointed restarts (a single long-running fetch process does not reliably survive a full ~75-minute run in this environment); `fetch_candidate_periodic_reports` was rewritten to checkpoint after every committee to a `.partial.csv` and resume automatically, rather than losing all progress on an interruption — this is now a permanent, tested feature of the fetch, not a one-off workaround.

**`scripts/estimate_candidate_spend_trickle.py` surfaced a real finding that overturned its own a priori design choice.** The script's original docstring reasoned that *median* would be the more defensible per-tier trickle-rate estimator (candidate disbursements are lumpy, so a few large periodic reports could dominate a mean). Checked against the real 2022/2024 panel, this was wrong for a different reason than anticipated: **median\_rate\_per\_day is exactly $0.00/day in every tier**, not because campaigns aren't spending, but because FEC's quarterly filing cadence, compared against this project's biweekly period grid, means the large majority of 14-day windows contain no new filing at all — a delta of zero between filings, not zero growth. Mean is the estimator actually used now (`preferred_estimator` in the saved JSON), since it correctly reflects the real jumps at filing dates as an average daily rate — the quantity `solve_bellman_lsm.py`'s wait branch actually needs. Real fitted rates (2022+2024 pooled, $/day): Safe D \$3,951, Likely D \$6,787, Lean D \$12,236, **Toss-Up \$12,725**, Lean R \$9,453, Likely R \$5,355, Safe R \$631 (pooled \$4,359).

§7.1's four-part self-consistency gate (`scripts/simulate_and_validate.py`) still passes unchanged — it does not exercise the new trickle mechanism directly (see §12.4's stated scope boundary on this), so an unchanged pass here confirms no regression, not a validation of the new mechanism specifically.

**A fourth correction, found the same way the first three were — by checking the mechanism before trusting the number, not after.** The first full production run (`scripts/solve_bellman_lsm.py`, $K=2000$) produced $\Theta(0)$ figures of roughly $+6.8$ to $+8.0$ expected seats — an order of magnitude larger than anything this research line had produced before, and `frac_deploy_now` collapsing to exactly $0.000$ at every period except the last. That size of jump was treated as a signal to check the mechanism, not a result to report. The cause: `_deploy_value()`'s "deploy now, then let remaining drift resolve" convolution (`Phi((mu_t+delta_mu)/sqrt(sigma_i^2+V(t)))`) is only valid when the *future* movement being integrated over is mean-zero — true of idiosyncratic $\varepsilon$ by construction, and, pre-trickle, trivially true of $D_{i,t}$ (it never moved, so it had no mean to be nonzero). Now that $D_{i,t}$ has a real, deterministic, non-zero-mean trickle, evaluating the deploy branch at today's $\mu_t$ plus only the DCCC's own increment — widened by idiosyncratic $\sigma$ alone — silently omitted the deterministic $\mu$ appreciation the candidate's own future organic spending produces between now and Election Day *regardless of today's decision*. The "wait" side of the recursion did not have this gap, because it is fit against real simulated future `mu_paths` that already reflect the grown $D$ — so the asymmetry made "wait" look better for a reason having nothing to do with genuine option value.

**Fixed**: `_deploy_value()` now also computes the deterministic terminal floor $D_{\text{terminal}}$ (known in advance — the trickle is deterministic) and an expected terminal $R$ ($r_t + \eta\cdot(D_{\text{terminal}}-D_t)$, the mean-zero residual component correctly left to `widened_sigma`, unchanged), recomputes the structural $\mu$ at that terminal pair, and adds the resulting shift on top of $\mu_t+\Delta\mu$ before convolving. This is not a stylistic nicety — it is the same class of fix as $\S$8.5's `beta1_open` correction (an inconsistency between two branches of the same computation), caught here before a single number was published rather than after, per this project's improving practice of writing the check before trusting the output.

**Results after the fix** (both the live 98-day-out run and the 1-year, 364-day-out counterfactual — the latter reuses `solve_bellman_lsm.run_lsm()` unmodified via monkey-patched `TODAY`/`N_PERIODS` globals, so it inherits the fix automatically):

| Scenario | Live $\Theta(0)$, pre-fix (§8.5) | Live $\Theta(0)$, corrected | 1yr $\Theta(0)$, pre-fix | 1yr $\Theta(0)$, corrected |
|---|---|---|---|---|
| `eta_fit_2022` | $-0.633$ (deploy 100%) | **$+2.202$** (deploy 0.0%) | $-0.746$ (deploy 100%) | **$+3.611$** (deploy 0.0%) |
| `eta_fit_2024` | $-0.109$ (deploy 100%) | **$+2.926$** (deploy 0.0%) | $-0.174$ (deploy 100%) | **$+3.822$** (deploy 0.0%) |
| `eta_bootstrap_all_cycles` | $-0.289$ (deploy 100%) | **$+2.489$** (deploy 0.0%) | $-0.447$ (deploy 100%) | **$+4.522$** (deploy 0.05%) |

**This is the first result anywhere in this research line where $\Theta$ is unambiguously, substantially positive and the corner flips to unanimous *wait*, not deploy.** Every prior finding — the original run, the $\hat\eta$/$G_t$ wiring fix, the `beta1_open` correction, the continuous-$\phi$ generalization — landed on "deploy now" by a comfortable-to-overwhelming margin. This reverses it, by roughly the same margin in the other direction (2–4.5 expected seats, not a knife-edge). Two features of the pattern are worth stating plainly rather than glossed over:

1. **$\Theta(0)$ is *larger* at the 364-day horizon than at the live 98-day horizon, in every scenario — wait is favored *more* strongly the further out the decision is made.** This is the same *direction* §7 of this document originally found pre-trickle (deploy-now got more strongly favored at longer horizons), but the *polarity* of what's being reinforced has flipped: previously a longer horizon reinforced the case for deploying immediately; now it reinforces the case for waiting. This new direction is also the one that matches real campaign behavior — §10's finding that real IE committees spend under 5% of their money by September 1 and 95%+ in the final two months is a report of committees *waiting far more the further out the decision is*, exactly what the corrected model now predicts, where the pre-fix model predicted the opposite and was flagged in §10 as contradicting observed practice. This is a genuine, if informal, cross-check the corrected model now passes that the prior model failed.
2. **This does not mean the "solver-consistent comparison" against Paper II's 65.5% baseline (§8.1's original criterion) can be run the same way as before.** That comparison assumed $\Theta(0)$ recommends deploying the reserve (in which case, apply that decision via the nonlinear optimizer and compare its allocation shares to the baseline). $\Theta(0)$ now recommends the opposite — hold the entire reserve — which is not a comparable "allocation" to score against a tier-share baseline; it is a recommendation to allocate nothing yet. Re-framing what the closing empirical deliverable should even measure, now that the answer is "wait," is flagged here as open (§12.5), not resolved by extension of the old framework.

`outputs/pre_dated_spend_fix/` holds the pre-fix snapshots (`theta_schedule.json`, `theta_schedule_1yr_counterfactual.json`) for the historical record, matching the `outputs/pre_beta1_open_fix/` precedent.

### 12.4 Scope boundaries, stated plainly

- **$\hat\eta$'s application to a candidate-committee spending trickle (rather than the IE-to-IE reaction it was estimated on) remains an untested extension of an existing estimate**, exactly as flagged when the mechanism was first wired in (§12's own note in `solve_bellman_lsm.py`, mirroring the $\alpha_3$ scope boundary already established in Paper III §5.5). Nothing in this section's re-run validates that extension directly.

2. Consider whether a cycle-weighted (DerSimonian-Laird, mirroring §6's treatment of $\hat\eta$) trickle rate would improve on the naive 7-cycle pool computed in §12.7 below, the same way cycle-weighting corrected $\hat\eta$'s naive pool.

| Pooled | 4,359 | 3,332 | −24% |

This is the same mechanism §5/§6 found for $\hat\eta$: a naive pool of only 2 cycles lets whichever cycle happened to have the most/largest filings dominate, and 2022/2024 appear to be higher-spend recent cycles relative to the 2012–2020 average, not representative of the full history. The direction of the correction (smaller, not larger, trickle rates) is the conservative one — it does not manufacture the reversal in §12.3; it moderates its magnitude, which is the honest expectation for a coefficient re-estimated on ~3.4x more data ($n=57{,}763$ delta-panel rows vs. $16{,}818$).

**The magnitude shrinks (roughly 40–55% smaller at the live horizon, matching the trickle rate's own shrinkage), but the qualitative finding is completely unchanged: $\Theta$ is positive in every scenario at both horizons, and hold remains the near-unanimous winner.** The 364-day-out figure is still larger than the 98-day-out figure in every scenario, reproducing §12.3's "wait favored more the further out" pattern. `eta_bootstrap_all_cycles` picks up slightly more non-hold mass than before (1.1% at the live horizon, 2.8% at 1yr, vs. 0.0%/0.05% previously) — a small, real effect of the smaller, more precisely estimated trickle rate leaving marginally less room for hold to dominate, not a sign of instability.

### 12.6 The continuous-$\phi$ generalization, wired in and re-run: the corner flips there too (added 2026-07-20)

§12.4's open item — whether the trickle mechanism, once wired into `solve_bellman_lsm_continuous_phi.py`, would leave the continuous-deployment-fraction corner intact — is now closed. The same three changes made to `solve_bellman_lsm.py`'s binary framing were mirrored here: `_solve_committed_floor()` now takes the current (trickled) floor $d_t$ as a parameter instead of closing over the static `floor_arr`; `run_continuous_phi_lsm()` builds the same trickle-driven `d_paths`/eta-reactive `r_paths` as `run_lsm()`; and the same drift-correction fix applies — but only to the $g=n_{\text{grid}}-1$ (full-deployment) grid level's precomputed $\mu$, since that is the only one later convolved with `widened_sigma` to produce `absorbing_val`. Every other grid level is used directly as real current-period regression features and needs no correction, exactly as `mu_baseline` ($g=0$) needs none.

**Result, $K=2000$, 5-point grid, all three scenarios, live 98-day horizon, $t=0$:**

| Scenario | $V$(hold, $\phi=0$) | $V$(deploy, $\phi=1$) | Gap (hold − deploy) | Chosen fraction |
|---|---|---|---|---|
| `eta_fit_2022` | 231.949 | 230.550 | **+1.399 seats** | 100.0% hold |
| `eta_fit_2024` | 233.025 | 231.177 | **+1.848 seats** | 100.0% hold |
| `eta_bootstrap_all_cycles` | 231.969 | 230.432 | **+1.537 seats** | 92.15% hold, 1.5% full deploy, 6.35% scattered across intermediate fractions |

**The corner flips here too, unanimously in two of three scenarios, and by a comfortable margin — not a knife-edge — in all three.** This directly answers §12.4's open question: the trickle mechanism's reversal of the binary framing's corner is not an artifact of the binary framing's coarseness; a genuinely continuous, sequential, correctly-concave version of the same decision reaches the same qualitative conclusion. `eta_bootstrap_all_cycles` shows a small, genuine minority (7.85% of paths) preferring something other than a full hold — the first non-degenerate split in the continuous framing since the pre-`beta1_open`-fix bootstrap scenario (§9 of this document) — but "hold" remains the dominant, plurality-winning option by a wide margin in that scenario too, and every intermediate fraction (0.25/0.50/0.75) sits well below both corners' values in all three scenarios, reproducing the same step-shaped value-of-budget curve (dominated by both corners, rarely by anything in between) that §8/§8.4 of Paper III already documented under the pre-trickle model — the mechanism behind that shape (the LP allocator's knapsack degeneracy) is unrelated to the trickle fix and was expected to persist.

**11-point spot-check on `eta_fit_2024`** (the standard practice from `docs/theta_followup_plan.md` §1.3 item 6, applied here even though no bracket is anywhere near indifference this time): confirms the 5-point result almost exactly — $V$(hold)$=233.039$ vs. $233.025$ at 5 points (a 0.014-seat difference, well within Monte Carlo noise), unanimous 100.0% hold at both resolutions. The corner is not a grid artifact.

Pre-trickle snapshots (`outputs/theta_schedule_continuous_phi_*_5pt.json`, `*_11pt.json`) are preserved under `outputs/pre_dated_spend_fix/`, matching this document's established snapshotting practice. New tests: `tests/test_bellman_lsm.py::TestContinuousPhiTrickle` (four cases — runs without error with a real trickle file, a nonzero trickle measurably changes output, the drift-correction formula behaves as specified, and the terminal period correctly has zero drift).

§12.5's open item 3 ("wire the trickle mechanism into the continuous-$\phi$ script") is resolved by this section and removed from the open-items list above.

### 12.8 A second use for the trickle calibration, outside the Bellman machinery it was built for (added 2026-07-22)

The candidate-committee trickle rate this section calibrated for $\Theta$'s wait branch turned out to have a second, unrelated application: correcting a *static* (non-$\Theta$) question about Paper II's live 2026 run. Paper II §7.1 had already named "near-zero candidate-committee floor spending" as part of the mechanism behind that section's 65.5%-to-non-competitive-seats finding, but had no way to test it directly — this section's dated panel (§2.7) made that possible.

A naive "seat gain from optimal deployment" computed straight from today's (near-zero) floors gave an implausible **+27.4 expected seats** — next to Paper I's validated +5.34-seat finding (DCCC's actual 2024 allocation vs. the model optimum; FINDINGS.md §10.3 traces this coefficient's own correction history — +4.46 was an interim, pre-α₅-fix value, since superseded), this was a signal to check the mechanism, not a number to report, in the same spirit as this section's own drift-correction finding (§12.3). `scripts/solve_2026_optimal_forecast_floors.py` generalizes this section's D-side trickle rate to **both parties** (projecting only D forward while leaving R frozen would trade the old asymmetry for a new one) and re-solves Paper II's optimizer against the resulting more-mature floors. Result: **+15.75 expected seats** — a real reduction, but flagged as still likely an overestimate, since coordinated/IE spend is held flat (the component §10 already found to be the most extreme back-loader of all) rather than projected, and even the candidate-side forecast is a smooth average against spending known to be lumpy.

**The number to actually trust for "how much does optimal allocation buy" remains Paper I's validated backtest finding (+5.34 / +6.79 expected seats, 2024/2022 OOS), not either 2026 live estimate** — both `+27.4` and `+15.75` answer "optimal vs. deploying nothing further," a more extreme comparison than Paper I's "optimal vs. what DCCC actually chose." (Corrected 2026-07-22: this paragraph previously cited +4.46/+5.36, an interim pre-α₅-fix figure per FINDINGS.md §10.3 — not Paper I's final validated result.)
