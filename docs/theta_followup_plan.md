# Why $\Theta$ Recommended Full Deployment Today: Candidate Explanations and Implementation Plan

*Companion to Paper III §8.2. That section reports the finding — $\Theta(0)$ recommends deploying the entire $393M reserve immediately, in both $\eta$-brackets, so the closing comparison against Paper II's 65.5% baseline lands unchanged (>30% band). This document lays out what to test next, in priority order, with concrete implementation steps for each.*

---

## 0. A correction found while starting this document — test this first

While preparing to write up "guess #2" (the environment-asymmetry hypothesis) below, checking the actual code in `scripts/solve_bellman_lsm.py` turned up something that changes the priority order: **neither of the two calibrated mechanisms this session spent most of its effort on — $\hat\eta(\text{tier})$ and $\sigma_G(\Delta t)$ — actually affected the reported result.**

- `eta_by_tier` is passed into `run_lsm()` and stored in the output dict for reporting, but is never multiplied into anything inside the function. This is not a subtle bug — it is structurally impossible for $\eta$ to matter under the current "wait" branch design, because $D_{i,t}$ never changes while waiting (no discretionary spending happens), so $\eta\cdot\Delta D_{i,t}=\eta\cdot 0=0$ regardless of $\hat\eta$'s value. The "two $\eta$-brackets" reported in §8.2 differed only in `resid_std_by_tier` (the *residual* noise scale from each cycle's regression — opponent spending variation *unexplained* by reaction), not in $\eta$ itself.
- $G_t$ was never simulated in `solve_bellman_lsm.py` at all — no `g_paths`, no reference to `SIGMA_G_PER_SQRT_DAY`. This is *consistent* with §5.5's scope boundary (never fold $\Delta G$ into $\mu_i$, since $\alpha_3$ was never estimated for within-cycle GB movement) but it means the national-environment calibration was completely absent from this run, not just present-but-symmetric as an earlier explanation loosely suggested.

**What this means:** the "deploy now" result was driven entirely by two things — the idiosyncratic $\varepsilon_{i,t}$ decay process and opponent's *unexplained* residual spending noise — both symmetric, mean-zero processes. The two mechanisms most likely to introduce a genuine asymmetry or a real reactive cost to spending were never in play. This is arguably a more important thing to fix than either guess below, because until it's fixed, guesses 1 and 2 are being tested on an incomplete model of $P$.

### 0.1 Implementation steps

1. **Give $\eta$ something to react to.** The "wait" branch currently means "zero discretionary spending, ever, until a deploy decision fires." To let $\eta$ matter, the wait branch needs a small, non-discretionary baseline spending trickle it can react to — the most defensible source is real, ongoing candidate-committee disbursement growth (already tracked as `cand_d_total`, currently held perfectly fixed as the floor). Model the floor itself as slowly growing over the remaining periods (a simple rate estimated from the historical panel), and let $R_{i,t}$ react to *that* via $\eta(\text{tier})$, in addition to the residual noise already modeled.
2. **Wire $\hat\eta$ into the deploy branch too.** `_deploy_value()` calls the LP allocator (`optimize()`), which has no adversarial-response parameter at all — unlike `optimize_nonlinear()`, which already supports `eta` for exactly this purpose (§I of the full derivation doc; `allocator.py`'s `_reactive_r`). Either switch the deploy-branch evaluation to a fast approximation that includes the same reactive-$R$ mechanic, or add an equivalent linear adjustment to the LP formulation (reduce the effective MSG gradient by a factor reflecting $\eta$, matching `_msg_vec`'s `eta`-adjusted gradient in the nonlinear path).
3. **Bring $G_t$ into the simulation, respecting the scope boundary.** The boundary says: don't apply $\alpha_3$ to within-cycle $G_t$ movement in the *margin model*. It does not say $G_t$ can't affect the *basis features* or the *value comparison* directly. One option: add $G_t$ (or its realized path) as a fifth regression-basis feature, so the continuation-value regression can learn a $G_t$-dependent adjustment empirically, without ever touching $\mu_i$'s structural formula. This sidesteps the boundary rather than violating it.
4. **Re-run the full $K=2000$, two-bracket LSM with both fixes applied**, and check whether $\Theta(0)$ changes. If it still favors full deployment, that is a stronger, more complete result than the current one (since it would no longer be attributable to two calibrated mechanisms sitting idle). If it changes, the current §8.2 finding needs to be revised, not just footnoted.

## 1. Guess A: the binary deploy/hold framing is too coarse

### 1.1 The hypothesis

The current backward induction only ever compares two extremes at each period: deploy the entire remaining $F_t$, or hold all of it. A real committee's actual discretion is continuous — spend some fraction now, reserve the rest. It is possible that "hold 100%" genuinely loses to "deploy 100%" today, while some smaller reserve (say 20-30%) would beat both, and the current framework simply cannot express that middle option, so it never gets to win the comparison.

### 1.2 Implementation steps

1. **Generalize the control to a discretized deployment fraction.** Replace the binary `deploy_vals[k] = _deploy_value(...)` / `wait_vals` comparison with a grid search over $\phi\in\{0, 0.25, 0.5, 0.75, 1.0\}$ (5 candidates is a reasonable starting resolution; can be refined later) at every (path, period).
2. **Extend `_deploy_value()` to accept a fraction.** Allocate $\phi\cdot F_t$ via the LP optimizer (`party_budget=phi*F_t` instead of `F0`), apply the resulting $\Delta\mu_i$ and widening exactly as now.
3. **Extend the continuation-value regression to condition on remaining budget.** Right now $F_t$ is implicitly always either $F_0$ (untouched) or $0$ (fully deployed) at the point of comparison. With partial deployment, remaining budget becomes a genuine state variable that varies across paths and periods — add it as a fifth feature to the compressed basis (alongside $E[\text{Seats}]_t$, $\text{Var}[\text{Seats}]_t$, $\text{MaxMSG}_t$, near-threshold count).
4. **Backward induction now solves a small optimization at each (path, period)**: for each of the 5 candidate $\phi$ values, compute deploy-now value (at that fraction) plus the regression-estimated continuation value (evaluated at the *reduced* remaining budget $(1-\phi)F_t$); take the best $\phi$. This is roughly 5x the optimizer calls of the current binary version — at $K=2000$, ~8 periods, 2 brackets, that's on the order of 250,000 LP calls, still well within the ~11ms/call budget (roughly 45 minutes total), not a barrier.
5. **Report the optimal $\phi^*(t)$ schedule**, not just a binary decision — this is a more informative output than the current binary one regardless of what it finds, and directly answers "how much should the campaign hold back today," not just "should it hold back everything or nothing."

## 2. Guess B: the current environment may be asymmetrically favorable

### 2.1 The hypothesis

At D+5.02 on the generic ballot, the current environment is unusually good for Democrats relative to most of the historical range in the four-cycle series (2018 D+8.6, 2020 D+7.0, 2022 D$-$1.0, 2024 D$-$1.2 — recall these are *final* cycle values from Paper I's `generic_ballot_by_cycle.csv`; the relevant comparison here is against the *historical daily trajectory's* typical range, not just the four endpoints). If the true process is mean-reverting toward a lower long-run level rather than a pure zero-drift random walk, waiting risks a one-sided loss (the environment fading back toward its historical average), which would argue *for* locking in current conditions by deploying now — but this is a directional effect a symmetric, zero-drift process cannot express at all, whichever direction it points.

**Note the honest complication already surfaced in §0**: since $G_t$ was not in the tested model at all, this hypothesis cannot yet explain the *actual* result reported in §8.2 — it can only be tested once §0's fix (bringing $G_t$ into the analysis) is done. Sequence this after §0, not instead of it.

### 2.2 Implementation steps

1. **Fit an Ornstein-Uhlenbeck model *with drift* to the pooled four-cycle historical series**, not just its volatility (§5.3 already fit the volatility term structure but explicitly deferred formal RW-vs-OU model selection). Regress $\Delta G_t$ on $(t\text{-time})$ and current level $G_t$: the standard discretized-OU regression is $G_{t+\Delta}-G_t = \kappa\Delta(\bar G - G_t) + \text{noise}$, i.e. regress $\Delta G$ on $G_t$ (with a constant) to recover $\kappa$ and $\bar G$.
2. **Check where D+5.02 sits relative to the fitted $\bar G$.** If $\bar G$ is meaningfully below +5.02, quantify the implied expected drift over the remaining ~98 days: $\mathbb E[\Delta G] \approx \kappa\cdot(\bar G - 5.02)\cdot(\text{days remaining})$.
3. **Re-run the LSM (after §0's fix) with this asymmetric, mean-reverting $G_t$ process** in place of a zero-drift assumption, and check whether $\Theta(0)$ or the optimal deployment fraction (if §1 is also implemented) changes, and in which direction. Note the direction is not obvious in advance: a expected-downward drift could *strengthen* the case for deploying now (lock in today's favorable conditions before they fade) rather than weaken it — this hypothesis needs to be tested, not assumed to point toward more patience.
4. **Sanity-check against the historical series directly**: for each of the four historical cycles, identify the point (if any) where that cycle's trajectory was at a similarly favorable level to today's +5.02, and check what actually happened to the environment between that point and Election Day in that cycle. With only 4 cycles this is illustrative, not a statistically powered test, and should be reported as such.

## 3. Suggested order of work

1. §0's fix (wire $\eta$ and $G_t$ into the model that's actually being tested) — this is a prerequisite, not an alternative, to §1 and §2, since right now neither guess is being tested against a complete $P$.
2. §2.1–2.2's OU-with-drift fit — cheap (reuses existing historical data, no new simulation) and informs whether §0's $G_t$ integration should include a drift term from the start.
3. §0's full re-run (binary framing, but now with $\eta$ and $G_t$ live) — check if $\Theta(0)$ changes before adding the extra complexity of §1's continuous framing.
4. §1's continuous-deployment-fraction generalization, on top of §0's corrected model — the more expensive change, worth doing once the simpler fix's result is known.

Each step should be reported honestly regardless of outcome, per Paper III §8.1's standing commitment — including if §0's fix alone reverses the "deploy now" finding, which would mean the current §8.2 result needs to be corrected in the paper, not merely supplemented.

## 4. Implementation log — what was actually done, and one item found to be blocked

**§2.1–2.2 done first** (`scripts/estimate_gb_ou_drift.py`), per §3's ordering, since it informs whether §0's $G_t$ integration needs a drift term from the start. Pooling all four historical cycles at a 14-day lag: $\hat b=-0.00312$ (SE $0.00349$, $p=0.372$) — **not statistically distinguishable from zero**. $\hat\kappa=0.00022\,\text{day}^{-1}$ ($\hat\tau\approx4491$ days — implausibly slow mean reversion, i.e. no real reversion signal at this timescale), $\hat{\bar G}=+4.05$ vs. today's live $+5.02$ (gap $+0.97$). Implied $\mathbb E[\Delta G]$ over the remaining 110 days: $\kappa(\bar G-G_{\text{today}})\times110\approx-0.02$ points — negligible next to $\sigma_G\approx2$ points at that horizon. **Conclusion: no meaningful drift signal exists at the live decision's horizon, confirming §5.3's finding, so §0's $G_t$ integration uses a zero-drift random walk**, not an OU-with-drift process. The four-cycle sanity check (illustrative only, not powered): 2018 and 2020 moved further favorable to Democrats from a similarly-favorable starting point (+4.22, +1.91 over ~575–585 days), while 2022 swung sharply the other way ($-5.54$ over 586 days) from a nearly identical starting level (+4.31) — direction is genuinely mixed across the only four data points available, consistent with the OU fit's non-significant drift and *against* treating guess B's directional hypothesis as resolved in either direction. `outputs/gb_ou_drift_fit.json` is the saved result.

**§0.1.2 done** (`scripts/solve_bellman_lsm.py`): `margin_gradient()` now takes `eta` and applies the $(1+\eta)/T$ adjustment (matching `allocator.py`'s `_msg_vec`), and `_deploy_value()` now passes `eta_arr` (built from `eta_by_tier`, previously computed and stored but never multiplied into anything) into every gradient call. This directly fixes the bug §0 found: $\hat\eta$ now discounts the deploy-branch's linearized MSG for every race, tier-specific.

**§0.1.3 done**: $G_t$ is now simulated in the wait branch as a standalone zero-drift random walk at $\sigma_G(\Delta t)$ (matching `simulate_and_validate.py`'s construction) and added as a fifth continuation-value regression feature (`g_t`, alongside the existing four). It is **not** wired into $\mu_i$'s structural formula, per §5.5's scope boundary — it is a state descriptor the regression can use, not a new argument to the margin model. Expected result under this specification: since nothing in $P$ actually depends on $G_t$ (by design, respecting the boundary, and confirmed by §2.1–2.2's near-zero drift), the fitted coefficient on `g_t` should come back close to zero and statistically unremarkable — a correctly-wired but currently-null feature, not evidence the fix did nothing. Reported alongside the re-run below rather than assumed in advance.

**§0.1.1 found to be blocked, not implemented.** Giving $\eta$ something to react to on the *wait* branch (as opposed to the deploy branch, which is now fixed) requires a non-discretionary baseline spending trickle — the plan's own proposal was to model candidate-committee floor growth (`cand_d_total`) from a rate "estimated from the historical panel." Checking `data_catalog.md` §2.1 and the raw files directly: `candidate_disbursements_{cycle}.csv` contains only `TTL_DISB`, a single cycle-cumulative-final total per candidate, with **no per-filing date field anywhere in this repository** — not sparse, genuinely absent. This is the identical, already-documented gap in `dynamic/ledger.py`'s `RealizedSpendCommitmentSource` docstring for coordinated expenditures ("cycle-cumulative only; this repo has no per-filing date source for this component"). There is no dated panel to fit a growth *rate* from, at any resolution. Per this project's own standing discipline (stated most explicitly in that same docstring: not fabricating "a synthetic dataset dressed up to look like real... data, which would misrepresent data provenance"), this item is left unimplemented and documented here as a real, likely-permanent constraint, rather than papered over with an invented pacing curve. **Consequence**: $\eta$ still cannot fire on the wait branch — $D_{i,t}$ is genuinely fixed while waiting, for a real data reason, not an oversight. The deploy-branch fix (§0.1.2) is the part of §0 that was actually achievable and is now in place.

**Re-run completed** (both eta-brackets, $K=2000$, `outputs/theta_schedule.json`). One implementation bug surfaced and was fixed along the way: `sm.add_constant()`'s default `has_constant="skip"` silently dropped the intercept column at $t=0$, since $G_0=0$ for every path (a deterministic, zero-variance column that `add_constant` mistook for an already-present constant) — shrinking the design matrix from 6 to 5 columns and crashing on `cont_fit.params[5]`. Fixed by passing `has_constant="add"` explicitly; re-run confirmed deterministic (identical numbers on both attempts up to the crash point) and produced finite results everywhere, including $t=0$'s degenerate case (`g_t_coef=0.0`, `p=nan`, expected since every path is identical at $t=0$ before any stochastic shock fires — not a new bug).

**Results**:

| | $t=0$ mean $\Theta$ | $t=0$ frac(deploy now) | Pattern across periods |
|---|---|---|---|
| eta_fit_2022 | $-0.517$ | 1.000 | Negative at every period ($-0.13$ to $-0.52$); no positive-patience window |
| eta_fit_2024 | $-0.039$ | 1.000 | Positive for 5 consecutive periods (84→28 days out: $+0.047,+0.097,+0.123,+0.076,+0.016$), negative at the 14-day and $t=0$ endpoints |

`g_t_coef` across all 14 period-fits: $|\hat\beta|<0.02$, $p\in[0.18,0.83]$ throughout — exactly the "correctly-wired but currently-null" result predicted above, confirming $G_t$ has no real channel to matter under the zero-drift, alpha3-respecting specification.

**Bottom line: the top-line "deploy now" recommendation is unchanged by the correction** — both brackets still recommend full deployment at $t=0$, so the §8.1 solver-consistent comparison against Paper II's 65.5% baseline still lands at >30%. But the correction was not vacuous: the 2024 bracket's $\Theta(0)$ moved from what the pre-correction run reported as a large negative number to $-0.039$ — an order of magnitude closer to zero — and its mid-cycle patience premium is now wider and somewhat stronger (peak $+0.123$ at 56 days out, vs. the pre-correction run's reported $+0.02$ to $+0.11$ over a vaguer "42–84 days out" window). This is reported in Paper III §8.2 and the Conclusion, superseding the pre-correction numbers there.

**What this means for §1 and §2, going forward**: §2 (environment asymmetry) is now substantially de-risked as an explanation for anything in this result — the OU-drift fit above found no significant drift at the live horizon, so a mean-reverting $G_t$ is very unlikely to change $\Theta(0)$ if implemented in full (§2.3's re-run remains open but is now a low-priority check, not a plausible source of a different top-line answer). §1 (the binary deploy/hold framing being too coarse to express a genuine small reserve) is now the most promising remaining lead — the 2024 bracket's $\Theta(0)$ sitting at $-0.039$ (much closer to indifference than $-0.517$) is exactly the situation where a continuous $\phi\in[0,1]$ framing could plausibly reveal a small nonzero optimal reserve that the current binary comparison cannot express. This is the natural next step, not yet implemented in this pass.
