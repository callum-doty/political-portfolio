This document lays out what to test next, in priority order, with concrete implementation steps for each.*

## 0. A correction found while starting this document — test this first

While preparing to write up "guess #2" (the environment-asymmetry hypothesis) below, checking the actual code in `scripts/solve_bellman_lsm.py` turned up something that changes the priority order: **neither of the two calibrated mechanisms this session spent most of its effort on — $\hat\eta(\text{tier})$ and $\sigma_G(\Delta t)$ — actually affected the reported result.**

- `eta_by_tier` is passed into `run_lsm()` and stored in the output dict for reporting, but is never multiplied into anything inside the function. This is not a subtle bug — it is structurally impossible for $\eta$ to matter under the current "wait" branch design, because $D_{i,t}$ never changes while waiting (no discretionary spending happens), so $\eta\cdot\Delta D_{i,t}=\eta\cdot 0=0$ regardless of $\hat\eta$'s value. The "two $\eta$-brackets" reported in §8.2 differed only in `resid_std_by_tier` (the *residual* noise scale from each cycle's regression — opponent spending variation *unexplained* by reaction), not in $\eta$ itself.
- $G_t$ was never simulated in `solve_bellman_lsm.py` at all — no `g_paths`, no reference to `SIGMA_G_PER_SQRT_DAY`. This is *consistent* with §5.5's scope boundary (never fold $\Delta G$ into $\mu_i$, since $\alpha_3$ was never estimated for within-cycle GB movement) but it means the national-environment calibration was completely absent from this run, not just present-but-symmetric as an earlier explanation loosely suggested.

**What this means:** the "deploy now" result was driven entirely by two things — the idiosyncratic $\varepsilon_{i,t}$ decay process and opponent's *unexplained* residual spending noise — both symmetric, mean-zero processes. The two mechanisms most likely to introduce a genuine asymmetry or a real reactive cost to spending were never in play. This is arguably a more important thing to fix than either guess below, because until it's fixed, guesses 1 and 2 are being tested on an incomplete model of $P$.

### 0.1 Implementation steps

1. **Give $\eta$ something to react to.** The "wait" branch currently means "zero discretionary spending, ever, until a deploy decision fires." To let $\eta$ matter, the wait branch needs a small, non-discretionary baseline spending trickle it can react to — the most defensible source is real, ongoing candidate-committee disbursement growth (already tracked as `cand_d_total`, currently held perfectly fixed as the floor). Model the floor itself as slowly growing over the remaining periods (a simple rate estimated from the historical panel), and let $R_{i,t}$ react to *that* via $\eta(\text{tier})$, in addition to the residual noise already modeled.
2. **Wire $\hat\eta$ into the deploy branch too.** `_deploy_value()` calls the LP allocator (`optimize()`), which has no adversarial-response parameter at all — unlike `optimize_nonlinear()`, which already supports `eta` for exactly this purpose (§I of the full derivation doc; `allocator.py`'s `_reactive_r`). Either switch the deploy-branch evaluation to a fast approximation that includes the same reactive-$R$ mechanic, or add an equivalent linear adjustment to the LP formulation (reduce the effective MSG gradient by a factor reflecting $\eta$, matching `_msg_vec`'s `eta`-adjusted gradient in the nonlinear path).
3. **Bring $G_t$ into the simulation, respecting the scope boundary.** The boundary says: don't apply $\alpha_3$ to within-cycle $G_t$ movement in the *margin model*. It does not say $G_t$ can't affect the *basis features* or the *value comparison* directly. One option: add $G_t$ (or its realized path) as a fifth regression-basis feature, so the continuation-value regression can learn a $G_t$-dependent adjustment empirically, without ever touching $\mu_i$'s structural formula. This sidesteps the boundary rather than violating it.
4. **Re-run the full $K=2000$, two-bracket LSM with both fixes applied**, and check whether $\Theta(0)$ changes. If it still favors full deployment, that is a stronger, more complete result than the current one (since it would no longer be attributable to two calibrated mechanisms sitting idle). If it changes, the current §8.2 finding needs to be revised, not just footnoted.

## 1. Guess A: the binary deploy/hold framing is too coarse

*Rescoped 2026-07-17, superseding the original sketch below §1.1 — the first pass (grid search over $\phi$ with a single-shot LP call per grid point, no carried-forward state) was structurally insufficient in two ways caught before implementation started: it would have masked concavity behind a linear interpolation, and it silently assumed the unspent fraction could be forfeited rather than carried forward. Both are corrected in §1.3 below. Not yet implemented — this is the design, not a result.*

### 1.1 The hypothesis

The current backward induction only ever compares two extremes at each period: deploy the entire remaining $F_t$, or hold all of it. A real committee's actual discretion is continuous — spend some fraction now, reserve the rest. It is possible that "hold 100%" genuinely loses to "deploy 100%" today, while some smaller reserve (say 20-30%) would beat both, and the current framework simply cannot express that middle option, so it never gets to win the comparison. The `eta_fit_2024` bracket's $\Theta(0)=-0.039$ (§4, §8.2) — an order of magnitude closer to indifference than `eta_fit_2022`'s $-0.517$ — is the concrete, data-grounded reason to expect this generalization might actually locate a non-zero interior answer rather than just re-confirm the corner. §7's finding sharpens why: $\Theta(t)$ as currently defined measures the value of flexibility *net of* what the deploy branch's own variance-convolution already captures for free, and that "already captured" share depends on how much of $F_t$ is being committed — so a coarser or finer split of the same total commitment could plausibly move the answer even without changing $F_0$ itself.

### 1.2 Why a naive grid search over $\phi$ would not work

Two structural requirements, both missed by the original sketch, have to hold simultaneously or this generalization will silently reproduce the binary answer while looking continuous:

1. **The deploy-value-at-$\phi$ function must be concave in $\phi$, not linearly interpolated between the two corner values.** $\Theta_\phi=\phi\, V^{\text{deploy}}+(1-\phi)V^{\text{wait}}$ is a convex combination of two fixed scalars — maximizing a linear function over a compact interval $[0,1]$ is mathematically guaranteed to land on a boundary unless the function happens to be exactly flat. That formulation could never surface an interior optimum; it would just recompute the existing binary answer with extra steps. The genuine source of concavity is the shadow price of capital falling as more is committed: `margin_gradient()`'s $c\cdot(1/D-(1+\eta)/T)$ term (`solve_bellman_lsm.py:187`) is strictly decreasing in $D$ — the same mechanism already verified as the structural driver behind §7's variance-convolution finding. Capturing it requires actually re-solving the LP allocator at each candidate deployment level, not interpolating between endpoints.
2. **Unspent capital must be carried forward, not forfeited.** Since `margin_gradient` stays positive for essentially every realistic $(D,T)$ pair in this universe (races here are heavily outspent, so $D\ll T$ almost everywhere), permanently setting money aside and never spending it is always weakly dominated by spending it eventually — a one-time static split chosen once at $t=0$ and never revisited would therefore never rationally choose $\phi<1$. This has to be a genuine sequential (impulse-control) problem: the reserve $F_t$ becomes a state variable that future periods can still act on, not a single interpolated choice at $t=0$. In the finance analogy this is closer to a multiple-exercise Bermudan option than a single European-style choice.

### 1.3 Implementation steps

1. **Discretize the reachable budget grid.** $\mathcal F=\{0,0.25F_0,0.5F_0,0.75F_0,F_0\}$ as a starting resolution (5 points). Legal actions from state $f\in\mathcal F$ are $a\in\mathcal F$ with $a\le f$; the post-action state carried into the next period is $f-a$.
2. **Fix the floor-reset bug this design would otherwise inherit.** `_deploy_value()` currently hardcodes `d_t = floor_arr.copy()` (`solve_bellman_lsm.py:299`) — the *original* candidate floor, on every call, regardless of what a path has already deployed. Calling the LP repeatedly against this same fixed baseline would give the same linearized `msg_i` every time (the existing LP-vs-nonlinear investigation, §8.2, already established that `optimize()` treats `msg_i` as a frozen constant per call, with no within-call diminishing-returns mechanism — it degenerates to a greedy knapsack). Without this fix, re-running the LP at successive nodes would trace out a flat line, not a concave curve, and the whole exercise would silently fail to find anything but a corner for a reason having nothing to do with economics. **Required change**: `d_t` must become `floor_arr + cumulative_deployed_state`, so each successive call re-linearizes `margin_gradient` at the higher, already-topped-up $D$ — this is what makes the sequence of LP calls trace out the true concave value-of-budget curve via successive re-linearization, consistent with how the existing chain-rule approximation already works within a single call.
3. **Backward induction over the expanded state**, for each state $(t,\omega,f)$ and each legal action $a$:
   $$V_t(\omega,f)=\max_{a\in\mathcal A(f)}\Big[\text{LP\_Alloc}(a,\mathbf X_t(\omega),\ \text{floor}=\text{floor}_0+F_0-f+a)+\hat{\mathbb E}_t\big[V_{t+1}(\omega,f-a)\big]\Big]$$
4. **Continuation regression, conditioned on remaining budget.** Two options, not mutually exclusive:
   - *Unified*: add $f'$ and $(f')^2$ as a sixth and seventh feature to the existing compressed basis ($E[\text{Seats}]_t$, $\text{Var}[\text{Seats}]_t$, $\text{MaxMSG}_t$, near-threshold count, $G_t$), one joint regression per period. Preferred as the starting approach — pools information across budget states, which matters because endogenous action choice (rather than uniform grid sampling) will likely leave some $(t,f)$ cells thin, especially in early periods before paths have had a chance to spread across reserve levels.
   - *Grid-stratified*: a separate regression per reachable $f'$ per period — avoids imposing a quadratic functional form on the budget/political-feature interaction, at 5x the (computationally trivial) regression cost. Use as a diagnostic check on the unified fit's residuals, not the default, given the sparse-cell risk above.
5. **Compute cost, using this project's actual $N_{\text{periods}}=7$ (live run), not an illustrative $T=15$.** Sum of legal actions across a 5-point grid is $1+2+3+4+5=15$ per (path, period); at $K=2000$: $15\times2000\times7=210{,}000$ LP calls $\times\sim$11ms $\approx$ 38 minutes per bracket. This is a batch job, not a quick script, but tractable.
6. **Grid-resolution sensitivity is not optional, given how close to indifference the target is.** A 5-point (25%-step) grid could report a corner as optimal purely from resolution if the true interior optimum is narrow (e.g. a 5–10% reserve) — indistinguishable, from the output alone, from a genuine bang-bang result. Run the 5-point grid first; if it reports anything other than a clean corner, re-run at an 11-point (10%-step) grid to confirm the interior answer isn't a grid artifact. Cost scales roughly as the triangular sum of grid points, so 5→11 points is roughly a 4.4x multiplier ($\approx$170 minutes per bracket) — worth budgeting for upfront rather than discovering after a coarse run comes back at a corner with no way to tell if that's real.
7. **Report the optimal $\phi^*(t)$ schedule** (or, more precisely, the optimal action $a^*$ from each reachable state), not just a binary decision — this is a strictly more informative output than the current binary one regardless of what it finds, and directly answers "how much should the campaign hold back today," not just "should it hold back everything or nothing."

## 2. Guess B: the current environment may be asymmetrically favorable

### 2.1 The hypothesis

At D+5.02 on the generic ballot, the current environment is unusually good for Democrats relative to most of the historical range in the four-cycle series (2018 D+8.6, 2020 D+7.0, 2022 D$-$1.0, 2024 D$-$1.2 — recall these are *final* cycle values from Paper I's `generic_ballot_by_cycle.csv`; the relevant comparison here is against the *historical daily trajectory's* typical range, not just the four endpoints). If the true process is mean-reverting toward a lower long-run level rather than a pure zero-drift random walk, waiting risks a one-sided loss (the environment fading back toward its historical average), which would argue *for* locking in current conditions by deploying now — but this is a directional effect a symmetric, zero-drift process cannot express at all, whichever direction it points.

**Note the honest complication already surfaced in §0**: since $G_t$ was not in the tested model at all, this hypothesis cannot yet explain the *actual* result reported in §8.2 — it can only be tested once §0's fix (bringing $G_t$ into the analysis) is done. Sequence this after §0, not instead of it.

### 2.2 Implementation steps

1. **Fit an Ornstein-Uhlenbeck model *with drift* to the pooled four-cycle historical series**, not just its volatility (§5.3 already fit the volatility term structure but explicitly deferred formal RW-vs-OU model selection). Regress $\Delta G_t$ on $(t\text{-time})$ and current level $G_t$: the standard discretized-OU regression is $G_{t+\Delta}-G_t = \kappa\Delta(\bar G - G_t) + \text{noise}$, i.e. regress $\Delta G$ on $G_t$ (with a constant) to recover $\kappa$ and $\bar G$.
2. **Check where D+5.02 sits relative to the fitted $\bar G$.** If $\bar G$ is meaningfully below +5.02, quantify the implied expected drift over the remaining ~98 days: $\mathbb E[\Delta G] \approx \kappa\cdot(\bar G - 5.02)\cdot(\text{days remaining})$.
3. **Re-run the LSM (after §0's fix) with this asymmetric, mean-reverting $G_t$ process** in place of a zero-drift assumption, and check whether $\Theta(0)$ or the optimal deployment fraction (if §1 is also implemented) changes, and in which direction. Note the direction is not obvious in advance: a expected-downward drift could *strengthen* the case for deploying now (lock in today's favorable conditions before they fade) rather than weaken it — this hypothesis needs to be tested, not assumed to point toward more patience.
4. **Sanity-check against the historical series directly**: for each of the four historical cycles, identify the point (if any) where that cycle's trajectory was at a similarly favorable level to today's +5.02, and check what actually happened to the environment between that point and Election Day in that cycle. With only 4 cycles this is illustrative, not a statistically powered test, and should be reported as such.

## 5. Reconciling two calibration-reliability caveats (added 2026-07-17)

Reviewing `outputs/eta_reaction_by_tier_fig.png` and `outputs/gb_asymmetry_check_fig.png`, two caveats already on record in Paper III §5.5 had never been resolved: $\hat\eta(\text{tier})$'s leave-one-cycle-out instability, and $\sigma_G$'s same-direction late-cycle realized moves sitting in apparent tension with this session's OU-drift fit (§0.1 above). Both are now reconciled with a formal check (`scripts/reconcile_eta_sigma_g_instability.py`), not just re-described.

**$\hat\eta(\text{tier})$ instability — first pass said "mostly noise"; that pass was underpowered, and a properly powered second pass says otherwise.** First pass: checking whether each tier's 2022 and 2024 95% CIs overlap (rather than comparing point estimates alone) found **5 of 7 tiers overlap**, including Toss-Up, Lean D, and Lean R — suggesting their large point-estimate swings (Toss-Up: 0.73 vs. 0.34) were honestly *unresolved*, not confirmed real. **A single pairwise comparison has very little power, so this was a falsely reassuring answer, not a validated one.**

Second pass (2026-07-17, same session, extending to all 7 cycles this repo has usable dated IE data for — 2012 through 2024; per §4.2's own data-quality table, the older cycles have *better* date completeness than 2022/2024, not worse): fitting eta per cycle and running one joint test per tier gives real statistical power, and the answer changes. **5 of 7 tiers show statistically significant cycle-to-cycle variation** (Safe D $p=0.002$, Lean D $p<0.001$, Toss-Up $p=0.001$, Likely R $p<0.001$, Safe R $p<0.001$) — including **Toss-Up and Lean D, the two tiers this project leans on hardest**. Toss-Up's single-cycle estimate actually goes negative in 2016 ($-0.22$), a sign flip the 2-cycle check could never have shown. Only Likely D ($p=0.06$) and Lean R ($p=0.22$, thinnest data of any tier) aren't confirmed to vary — plausibly still a power limitation for Lean R, not genuine stability.

**Net effect, superseding the first pass: this is not a narrow Safe-D/Safe-R problem — it is a whole-table problem.** Precision within a given 2-cycle pool (§4.4's activity-rate argument) is a different claim from temporal stability across cycles, and most tiers fail the stability test once tested with real power. The pooled point estimates remain the best available and are not withdrawn, but should be read as averages over confirmed real variation, not fixed constants. See Paper III §5.5's twice-updated Validation C.

**$\sigma_G$'s late-cycle bias — real direction, unreliable magnitude, not in conflict with the OU fit.** Extending Validation B's Sept-1-to-Election-Day check from 2 cycles to all 4 (2018, 2020, 2022, 2024): **all four cycles moved toward Republicans** in that window (not just the 2 originally checked), but the size of the move varies enormously — 2018/2020 were tiny ($z\approx-0.05$ to $-0.11$, indistinguishable from noise) while 2022/2024 were substantial ($z\approx-1.3$ to $-1.4$). Mean $z$ across all four is $-0.73$, materially smaller than the $-1.37$ the 2-cycle check alone suggested. This is **not in conflict** with the OU-drift fit's "no significant level mean-reversion" finding (§0.1) — the two ask different questions: the OU fit pools all lags across the full trajectory and asks whether $G_t$ reverts toward a fixed $\bar G$ regardless of calendar position; this check asks specifically about the late-cycle window. A weak, inconsistent-magnitude late-cycle drift can coexist with no detectable level mean-reversion overall. **Practical implication, if this pattern is real**: the consistent direction (toward Republicans, i.e. away from today's Democratic-favorable D+5.02) argues for *locking in* the current environment by deploying now, not for patience — reinforcing, not reversing, §8.2's "deploy now" finding, for a reason distinct from anything tested in §0 or §2's OU fit. Given the wildly inconsistent magnitude across only 4 data points, this should be read as a real but weak and unreliable signal, not a basis for a new calibrated drift term.

All reconciliations are saved (`outputs/eta_instability_reconciliation.csv`, `outputs/sigma_g_late_cycle_reconciliation.csv`, `outputs/eta_seven_cycle_extension.csv`, `outputs/eta_seven_cycle_joint_test.csv`) and reflected in Paper III §5.5.

## 6. From "eta is unstable" to a usable uncertainty-aware model (added 2026-07-17)

§5 established that $\hat\eta$ genuinely varies cycle-to-cycle for 5 of 7 tiers. This section turns that finding into two concrete deliverables, chosen from a wider brainstorm (hierarchical Bayesian, hidden regimes, state-space/Kalman, structural decomposition, robust optimization, and Bayesian model averaging were all considered and set aside — see rationale below) for being immediately buildable from data already on disk without imposing assumptions the data can't support.

**1. Two uncertainty-aware alternatives to the pooled point estimate** (`scripts/build_eta_uncertainty_distribution.py`, `data/processed/eta_uncertainty.json`):
- **Non-parametric bootstrap**: resamples each tier's 6–7 real per-cycle point estimates with replacement — no assumption about shape, which matters since the real data include a sign flip (Toss-Up: $-0.22$ in 2016) a fitted Normal would smooth away.
- **Random-effects meta-analysis (DerSimonian-Laird)**: separates real between-cycle variance ($\tau^2$) from each cycle's own estimation noise, reporting $I^2$ (share of cross-cycle spread that's real, not noise) and a predictive SD for an unseen cycle. $I^2$ independently reproduces §5's significance results: 0% for Lean R (matches its non-significant joint test), 70–89% for the five significant tiers.
- **A genuinely new finding surfaced by building this**: the properly cycle-weighted pooled means differ substantially from the naive pooled-panel numbers already in Paper III §4.4 — e.g. Lean D drops from 0.637 to 0.277, Safe R from 0.584 to 0.258 — because naive pooling lets whichever cycle generated the most transaction rows dominate, while cycle-weighting gives each *election* a vote scaled by its own precision. Paper III §4.4 now carries both figures with the cycle-weighted one flagged as the better estimate.

**2. Wiring a per-path draw into the Monte Carlo** (`scripts/solve_bellman_lsm.py`): `run_lsm()` was refactored to take `eta_arr_by_path`/`resid_std_arr_by_path` of shape `(K_PATHS, n)` instead of a single dict broadcast to every path, unifying the old two-bracket behavior (`tile_single_cycle()`, identical value tiled across all paths) and the new bootstrap behavior (`bootstrap_eta_resid_paths()`, each path draws its own per-tier $(\hat\eta, \text{resid\_std})$ pair from one randomly chosen historical cycle, held fixed for that whole path). A new `eta_bootstrap_all_cycles` scenario runs alongside the original two brackets, not replacing them. Result: $\Theta(0)=-0.414$, and `frac_deploy_now=91.5\%$ — the first *non-degenerate* $t=0$ statistic this project has produced, since paths under bootstrap genuinely differ from each other even before any period elapses (unlike the single-cycle brackets, where every path shares an identical pre-shock state and the 0%/100% split is closer to an artifact of that than a real distribution). Full detail in Paper III §8.2.

**Why the other six brainstormed options were set aside** (not because they're wrong, but because $n=5$–$7$ cycles per tier can't support them yet):
- *Hierarchical Bayesian (tier × cycle crossed random effects)* — the right long-run answer, and DerSimonian-Laird above is a single-tier special case of it; a full crossed model is more work for a modest gain at this sample size and is a natural next step, not implemented here.
- *Hidden regimes* — only identifiable from an externally pre-specified rule (midterm/presidential, a recognized wave-election definition), not from the eta values themselves (which would be circular); not attempted.
- *State-space/Kalman random walk* — hits the same wall the OU-drift fit already hit with far more data (§0.1: $p=0.37$ pooling thousands of points); with 5–7 points per tier, a transition variance isn't identifiable. The data itself argues against this framing too: Toss-Up's sequence ($+0.42,+0.25,-0.22,+0.08,+0.11,+0.73,+0.34$) bounces and snaps back rather than wandering and staying wandered, more consistent with noise-around-a-mean than a persistent random walk.
- *Structural decomposition* ($\eta=\eta_{\text{fundraising}}+\eta_{\text{media}}+\eta_{\text{polarization}}$) — no data in this repository attributes IE spending to a causal channel; a different research program, not an extension of this one.
- *Robust optimization* (max-min over $\eta\in[\eta_{\min},\eta_{\max}]$) — a genuinely different, assumption-light paradigm worth prototyping separately; not implemented here since it changes the optimizer's objective, not just the calibration inputs.
- *Bayesian model averaging* (pooled vs. 2022 vs. 2024) — redundant with the bootstrap/random-effects approach above: these were never really competing model specifications, just different subsets of the same data.

## 7. A 1-year-horizon counterfactual, and what it reveals about the model's structure (added 2026-07-17)

§6's `eta_bootstrap_all_cycles` scenario was the first to show a non-degenerate $t=0$ split (`frac_deploy_now=91.5\%`). A colleague reviewing that result asked the natural question: is 91.5% (rather than 100%) mostly a function of the live decision sitting only ~98 days from Election Day — i.e., would deploy-now fall substantially if the same model were run from a full year out, since more remaining time should mean more option value in waiting? A controlled counterfactual was built to test this directly (`scripts/solve_bellman_lsm_1yr_counterfactual.py`), and it did not confirm the prediction — it reversed it.

**Design.** The script monkey-patches `solve_bellman_lsm`'s module-level `TODAY`/`N_PERIODS` globals (`ELECTION_DAY - 364 days`, giving 26 biweekly periods instead of the live 7) before calling the same `build_universe()`, `run_lsm()`, `tile_single_cycle()`, and `bootstrap_eta_resid_paths()` functions, unmodified. **Explicitly not** a reconstruction of what the 2026 race universe looked like a year ago — §0.1.1 already established no dated candidate-committee floor panel exists to build that from. Instead, today's actual observed starting state (floors, $R_{i,t}$, $\sigma_i$) is held exactly fixed, and only the artificial "days until Election Day" parameter changes — isolating the pure time-horizon effect, at the cost of not being a literal history. Both this run's $t=0$ and the live run's $t=0$ are true, dispersion-free starting points (no epsilon has accumulated yet in either), so comparing $t=0$-to-$t=0$ across the two runs is a clean test; comparing any other matched days-remaining point between them is not, since `remaining_variance()` is anchored to absolute days-remaining and the 364-day run has already burned through much of its variance budget by the time it reaches, say, 98 days remaining mid-path.

**Results** (`outputs/theta_schedule_1yr_counterfactual.json`):

| | Live, 98 days out | 1-year counterfactual, 364 days out |
|---|---|---|
| eta_fit_2022 | $\Theta(0)=-0.517$, deploy=100.0% | $\Theta(0)=-0.628$, deploy=**100.0%** |
| eta_fit_2024 | $\Theta(0)=-0.039$, deploy=100.0% | $\Theta(0)=-0.122$, deploy=**100.0%** |
| eta_bootstrap_all_cycles | $\Theta(0)=-0.414$, deploy=**91.5%** | $\Theta(0)=-0.586$, deploy=**100.0%** |

**The prediction did not hold, in either direction it could have failed softly — it reversed outright.** The one scenario with any wait mass at the live horizon (bootstrap, 8.5% wait) goes to *full* deployment at the longer horizon, not less. This is a real, deterministic feature of the model, not noise (the Bellman recursion has no randomness in which branch wins conditional on the inputs already drawn), so it was traced to its mechanical source rather than left as "we don't know why."

**Mechanism, verified directly from the code, not inferred:**

- **Discounting is not the cause — there isn't any.** `V_star = np.where(deploy_now, deploy_vals, wait_vals)` (`solve_bellman_lsm.py:355`) carries no $\beta^t$ term anywhere in the recursion; the only `gamma` in the file is the LP allocator's risk-aversion coefficient (`allocator.py`), unrelated to time. There is no discount factor to vary between the two runs.
- **Deploying is not a persistent state improvement that compounds over more future periods — it's a one-shot absorbing action.** `_deploy_value()` sets `d_t = floor_arr.copy()` (`solve_bellman_lsm.py:299`) at every backward-induction step, always resetting to the *original* candidate floor. A path's earlier deploy decision never updates the state later periods see. This is optimal-stopping (American-option) structure, not a control problem where an early action keeps paying off.
- **The actual mechanism is the deploy branch's variance-convolution identity, and it is quantitative, not speculative.** The deploy branch evaluates $\Phi(\mu_i/\sqrt{\sigma_i^2+V(t)})$ rather than $\Phi(\mu_i/\sigma_i)$ — a closed-form application of $\mathbb E[\Phi((\mu+\xi)/\sigma)]=\Phi(\mu/\sqrt{\sigma^2+V})$ for $\xi\sim N(0,V)$ — specifically so that deploying now already analytically integrates over the effect of *all* remaining idiosyncratic drift between now and Election Day, in one step, rather than requiring the model to actually wait through it. $V(t)=\sigma^2(1-e^{-\lambda \cdot \text{days\_remaining}})$, using the fitted $\lambda=0.00536$/day, gives $V(t)/\sigma^2 = 41\%$ at 98 days remaining but $86\%$ at 364 days remaining. As the horizon grows, deploying-now's built-in approximation to "wait and see" captures a larger and larger share of the uncertainty that literal waiting would resolve — shrinking the residual, distinct value that actual sequential waiting still has left to add. That is what falls as the horizon lengthens, not the value of information itself.

**Sharper statement of the finding, and its limit:** the apparent reduction in option value with longer horizons is not a generic property of optimal stopping — it arises because the deploy branch analytically integrates over a growing fraction of future idiosyncratic uncertainty through the variance-convolution identity. As the remaining horizon increases, this approximation captures an increasing share of the uncertainty that explicit waiting would otherwise resolve (41% at 98 days vs. 86% at 364 days, under the fitted decay rate), reducing the incremental value of sequential observation. Taken to its limit, as $\text{days\_remaining}\to\infty$, $V(t)\to\sigma^2$ and the deploy branch integrates over almost the full uncertainty budget without giving up the ability to act today — a very favorable approximation for "deploy now" to hold. The convolution identity is mathematically correct for what it integrates over: the expected effect of unresolved Gaussian idiosyncratic shocks on the terminal win probability. **What it does not, and structurally cannot, capture is the value of adaptive decision-making** — i.e., that new information arriving mid-campaign could change *how* the reserve is allocated, not just *whether* it's deployed. In this model, deployment is a one-time absorbing decision and the allocation is computed once, at the moment of deployment (§0.1.1's data gap is exactly why the wait branch can't even let $\eta$ react to a non-existent spending trickle); that adaptive channel is intentionally absent from the model as built, which is precisely why the convolution can substitute for so much of waiting's value. Put plainly: this model solves an optimal-stopping problem with analytical expectation-substitution for future shocks, not a fully adaptive sequential control problem — and once that distinction is visible, the 1-year counterfactual's direction stops being surprising and becomes an expected consequence of the model's own construction. This is worth stating explicitly in Paper III as a scope boundary (alongside §0.1.1's other absorbing-decision caveat), not left implicit: any reader tempted to interpret $\Theta(t)$ as "the value of flexibility" in the fully general sense should read it instead as "the value of flexibility *net of* what the deploy branch's own analytical widening already captures for free" — a narrower and more accurate claim.

## 8. The continuous-deployment-fraction generalization, implemented and run: corners persist (added 2026-07-17)

§1.3's design is now implemented (`scripts/solve_bellman_lsm_continuous_phi.py`) and run at $K=2000$ across all three $\eta$ scenarios (`eta_fit_2022`, `eta_fit_2024`, `eta_bootstrap_all_cycles`), 5-point grid ($\mathcal F=\{0,0.25,0.5,0.75,1.0\}\times F_0$), then spot-checked at an 11-point grid (10%-step) per §1.3 item 6's sensitivity requirement. **The answer is no: allowing a genuinely continuous, sequential deployment fraction does not surface an interior optimum. Every scenario recommends full deployment today, unanimously across all simulated paths, at both grid resolutions.**

### 8.1 Implementation notes and two deliberate departures from §1.3's literal text

The floor-reset bug (§1.3 item 2: `_deploy_value()` hardcoding `d_t = floor_arr.copy()` regardless of how much has already been committed) is fixed by construction: `_solve_committed_floor()` computes the per-race allocation for a *target cumulative budget* $C_g$ via the same LP call `_deploy_value()` already makes, but genuinely parameterized by $C_g$ instead of frozen at $F_0$. This is precomputed once per (grid level, path, period) — $4\times2000\times8=64{,}000$ LP calls at 5 points, $10\times2000\times8=160{,}000$ at 11 points — rather than once per (starting-state, action) pair as §1.3 item 5's literal accounting assumed (which would have given $\sim210{,}000$ and $\sim170$-minute figures); collapsing the redundant recomputation this way is a genuine efficiency gain, not just a simplification, and kept both the 5-point ($K=2000$, ~12 min/scenario) and 11-point ($K=500$ spot-check, ~8 min) runs tractable in this session.

Two departures from §1.3's literal text, both made deliberately and both load-bearing for the result:

1. **Exact nonlinear valuation instead of a linearized chain-rule delta.** §1.3's original phrasing (written before implementation) suggested extending `_deploy_value()`'s existing pattern of computing a linearized `delta_mu = grad * delta_s` and adding it to `mu_t`. Implemented instead: once `_solve_committed_floor()` returns the resulting per-race floor, this script recomputes $\mu$ directly from the exact structural formula already used everywhere else in `solve_bellman_lsm.py` ($\alpha_0+\alpha_1\text{pvi}+\alpha_2\mathbb 1[\text{incumbent}]+\alpha_3 GB+c\cdot\log(D/(D+R))$), with $R$ increased by $\eta_i\cdot(D_{\text{new}}-D_{\text{floor}})$ — i.e. paper3_draft.md §4.1's own $R_i(D_i)=R_{\text{base}}+\eta\cdot\max(0,\text{party}_i-\text{party}_{i,\text{obs}})$ applied directly, rather than folded into a chain-rule slope. This is strictly more faithful to §4.1's stated specification than a linearized delta would have been, and it is what gives the value-of-budget curve genuine, non-approximated concavity ($\log(D/(D+R))$ is itself concave in $D$) — avoiding exactly the class of frozen-linearization bug this whole exercise exists to fix. The LP allocator's *choice* of which races receive money still uses the linearized, $\eta$-discounted MSG objective (§0.1.2), since the fast LP's objective must be linear — but valuing the resulting allocation does not have to inherit that approximation, and here it doesn't.
2. **Grid-stratified regression instead of the unified quadratic-in-remaining-budget basis §1.3 item 4 stated as preferred.** Discovered empirically while smoke-testing at small $K$: pooling all grid states into one regression with $f'$/$(f')^2$ as extra features produced visibly non-monotonic continuation-value predictions across budget levels — the fitted quadratic was a poor functional form for what turned out to be a genuinely *step-shaped* value-of-budget relationship (see §8.2), not the smooth curve the unified basis assumes. Switching to a separate, unconstrained OLS fit per grid state (no shape imposed across budget levels) fixed this immediately. §1.3's stated reason to prefer unified — sparse cells under endogenous action-sampling — does not apply to this implementation's design anyway: every grid state is evaluated densely for all $K$ paths at every period (mu_committed is precomputed for the full grid $\times$ full path set, not sampled along a single forward-simulated policy trajectory), so there is no sparse-cell risk to trade off against.

### 8.2 Why the value-of-budget curve is a step, not a smooth concave curve

The isolated mechanism (checked directly, independent of the backward induction): at $t=0$'s baseline state, $E[\text{Seats}]$ rises from 221.65 (nothing committed) to 225.64 at just 5% of $F_0$ committed, to 225.78 at 25%, to 225.81 at 100% — a huge jump in the first few percent of the budget, then near-total flatness. This traces directly to a mechanism already documented in paper3_draft.md §8.2's addendum: the LP allocator's objective (`maximize msg @ s`) is a pure greedy knapsack with no diminishing-returns mechanism within a call, so it funds the **same fixed set of $\sim$7 low-floor outlier races regardless of total budget** — larger budgets just mean those same 7 races receive proportionally more money each, which saturates $\log(D/(D+R))$ (since $D$ quickly dominates $R$ for these historically under-funded races) almost immediately. The 11-point grid confirms this at finer resolution directly: option values for 10% through 90% committed sit within 0.001 seats of each other (226.2531–226.2544), while both corners (hold, 226.3625; deploy-all, 226.4702) sit clearly above the entire interior range.

### 8.3 Results at $t=0$ (the live decision)

| Scenario | 5-pt: $t{=}0$ chosen fraction | $V(t{=}0)$, full deploy | $V(t{=}0)$, next-best option | 11-pt spot-check |
|---|---|---|---|---|
| `eta_fit_2022` | **100.0%**, unanimous (2000/2000 paths) | 226.352 | 225.839 (hold) | — |
| `eta_fit_2024` | **100.0%**, unanimous (2000/2000 paths) | 226.470 | 226.340 (hold) | **100.0%**, unanimous (500/500 paths); identical $V(0)$ to 5-pt |
| `eta_bootstrap_all_cycles` | **100.0%**, unanimous (2000/2000 paths) | 226.240 | 225.839 (hold) | — |

Every scenario is a clean corner by a comfortable margin — `eta_fit_2024`'s gap (0.130 seats between full-deploy and the next-best option) is the narrowest of the three, consistent with it being the bracket whose binary-framing $\Theta(0)=-0.039$ sat closest to indifference, but even there the margin is not close enough that grid resolution could plausibly flip it, and the 11-point check confirms this directly. Intermediate fractions (25%/50%/75% in the 5-point grid; 10–90% in the 11-point grid) are essentially tied with each other and dominated by *both* corners at every period in every scenario — never the argmax for more than a few percent of paths, and typically far less.

**The most notable single number is `eta_bootstrap_all_cycles` moving from a genuine 91.5% split (§6, under the original binary framing's linearized valuation) to a unanimous 100% here.** This is not a contradiction — it is a consequence of using a materially more exact valuation (§8.1 item 1) on a genuinely different, richer state space (a carried-forward budget variable, not a one-shot choice). It is a reportable, honest difference in the corrected model's answer, not evidence either run was wrong.

### 8.4 Interpretation: §1.1's motivating hypothesis is not confirmed

§1.1 asked whether the binary framing was masking a genuine interior reserve — plausible on its face, since `eta_fit_2024`'s $\Theta(0)=-0.039$ sat an order of magnitude closer to indifference than `eta_fit_2022`'s $-0.517$. Having now built the properly concave, sequential, carried-forward-budget version, **the answer is no**, and the reason is economically legible rather than a modeling accident: any partial commitment pays the full cost of foreclosing flexibility on the committed portion (money once given to the same 7 knapsack-selected races cannot be un-given, and doesn't buy access to a different, better opportunity later) without capturing the deploy branch's exclusive benefit — the variance-convolution identity (§7 above; paper3_draft.md §8.3) that analytically integrates over *all* remaining idiosyncratic drift in one step. Holding entirely preserves 100% of that flexibility; deploying fully captures 100% of the analytic-integration benefit; anything in between gets a diluted share of the first with none of the second's completeness. Given the LP's own knapsack degeneracy also means there is essentially nothing left to gain past the first few percent of the budget, a genuine interior optimum was never structurally available to find — not because the binary framing failed to express it, but because this model's actual economics don't produce one. This should be read together with, not as a contradiction of, §7's scope-boundary finding: $\Theta(t)$ here measures the value of flexibility *net of* the deploy branch's own analytical widening, and that same widening is precisely why fractional deployment can't compete with either corner.
